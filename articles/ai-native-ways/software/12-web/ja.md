# 社外に見せる窓 ── 静的サイトと Cloudflare Pages

ここまでは社内向けの道具だった。最後に、社外に見せる **窓** ── Web を
立てる。会社サイト、製品ページ、ドキュメント、ブログ ── その大半は、
**静的**でよい。

そして静的なら、**自分のサーバすら持たなくていい**。焼いた HTML を
**Cloudflare Pages** に載せれば、それで公開できる。

## なぜ静的か

社外サイトの中身は、めったに変わらない。だから毎回サーバで組み立てる必要は
なく、**あらかじめ HTML に焼いて置く**ほうが、速く・安全・安い。

- **速い** ── ただのファイル配信。データベースを待たない
- **安全** ── 動く部分がないから、攻撃面がほぼ無い
- **AI と相性がいい** ── Markdown を書けば、Claude が HTML に組む

> このサイト(aiseed.dev)自身が、その実例だ。Markdown を静的 HTML に
> 焼いて配っている ── 動的サーバは、無い。

## 静的サイトを作る

文章は Markdown で書き、**Astro** や **Hugo**(あるいは自前のビルド)で
HTML に組む。出力は、ただのファイル群だ。

```bash
npm run build        # 例:Astro。出力は dist/(または html/)に焼ける
```

中身の更新は「書いて、組んで、上げる」だけ ── ここでも、書くのはコードでは
なく文章だ。

## Cloudflare Pages に載せる ── サーバを持たない

焼いた HTML の置き場は **Cloudflare Pages**。**サーバは一台も要らない**。
ファイル群をアップロードすれば、世界中の CDN から配信され、**HTTPS は自動**、
DDoS 防御も付く。

これは第10章の送信中継や、後で触れる AI と同じ構図だ ── **配信と防御は
前段で借り、ソースとビルドは自分の側に残す**。公開する HTML には秘密は無い
(焼かれた成果物だけ)から、これは安心して借りられる層だ。

## ビルド・確認・デプロイを分ける

ここが、この章の構造的な肝だ。公開は **三つの独立したステップ**に分ける。

```bash
# 1. ビルド ── html/ を生成する
npm run build
# 2. ローカルで目視確認 ── 再ビルドも監視もしない(焼いた物がそのまま見える)
python -m http.server --directory html 8000
# 3. プレビュー URL へ ── 実プラットフォーム上で答え合わせ
wrangler pages deploy html --branch preview
# 4. 確認できたら、本番へ
wrangler pages deploy html --branch main
```

**自動再ビルドや「ビルド＋デプロイ一括」は使わない**。裏で勝手に作り直されると、
目で確認した物と、本番に上がる物がずれる。ビルドした瞬間の `html/` を凍結し、
それを確認し、**同じ物**をデプロイする。アップロードは wrangler でも、Pages の
直アップロード API でもよい ── 大事なのは、間に**確認**を挟むことだ。

> 確認した物と、本番に上げる物を一致させる。
> 「現行を正解器に、確かめてから切り替える」── それを、公開でもやる。

## ドメインをつなぐ

独自ドメインは、Pages の **カスタムドメイン**に追加するだけでよい。DNS を
Cloudflare で持っていれば、**今までサーバを指していた A レコードが、自動で
Pages に差し替わる**。証明書も自動で発行される。

**メール(MX・SPF)は触らない** ── Web の置き場を替えるだけで、メールの
経路は別物だ。切替が不安なら、旧サーバを止めずに **プレビューで確かめてから**
本番ドメインを向ける(第8〜11章と同じ、並行稼働での切替だ)。

## 動的が要る所だけ、つなぐ

問い合わせフォームのような **動く部分だけ**、裏につなぐ。送信先は、第7章の
門番の内側に置いた自分の API(認証は門番、記録は第6章の DB、通知は第10章の
メール)だ。静的な窓に、動的な点を最小限だけ足す ── **大半は静的、要る所
だけ動的**。

## 内部の道具は、別の入口

公開サイトは Pages に **借りる**。だが、第7〜11章で立てた **内部の道具**
(認証・文書・コード・会議)は別だ。秘密と生データが通るそれらは、第7章の
**門番＝自分のリバースプロキシの内側**に残す。

- **公開の窓**(静的・秘密なし)── Cloudflare Pages に借りる
- **内部の金庫**(認証・業務データ)── 自分の側、門番の内側に置く

**窓は借り、金庫は自分に**。どちらを借り、どちらを握るかを、分けて決める。

## まとめ

社外の窓を静的に、サーバ無しで。

- **静的サイト(Astro / Hugo / 自前)** ── Markdown を HTML に焼く、速く・安全・安い
- **Cloudflare Pages** ── サーバを持たず公開、CDN・自動 HTTPS・DDoS を前段で借りる
- **ビルド → 確認 → デプロイを分ける** ── 自動再ビルドを避け、確認した物と本番を一致させる
- **ドメインは自動差し替え** ── カスタムドメインで A レコードが Pages に向く、メールは触らない
- **窓は借り、金庫は自分に** ── 内部の道具は第7章の門番の内側に残す

社内の道具と、社外の窓が揃った。最後の章では、これらすべての上に
**AI(自前の LLM と RAG)** を乗せ、Copilot の依存を断つ。
