- 作成日: 2026年7月2日

## 1. 連載コンセプト

**一文で:** Fable 5 は「何にでも使うAI」ではない。高コストな最上位モデルを、"任せる価値のある仕事" にだけ使うための見極め方を伝える連載。

### 背景となる前提

- Fable 5 は Anthropic の最上位「Mythosクラス」の初の一般提供モデル。長く複雑な仕事を、途中で文脈を失わずに自律的に進めることを得意とする。
- API料金は Opus 4.8 の2倍、Sonnet 4.6 の3倍以上。日常的な質問や短い文章作成に使うのはコスト的に不合理。
- したがって読者に必要なのは「Fable 5 のすごさの紹介」ではなく「どの仕事を Fable 5 に回し、どの仕事を Sonnet / Opus で済ませるか」という実務的な判断基準。

### 連載を貫く問い

> その仕事は、Fable 5 に頼む価値があるか?

各回の末尾で必ずこの問いに立ち返る(→ 執筆ガイド「記事の型」参照)。

### 連載の裏テーマ(全回に通底する原則)

> **方向性の決定は人間、実現はAI。**

Fable 5 が引き上げたのは「実現力」の上限であり、指示が薄いほど出力は平均的・既視感のあるものに収束する。デザインなら参考イメージ、文章なら視点と体験、コーディングなら設計方針 — 人間が渡す「核」の質が出力の質の上限を決める。第1回で初出させ、第4回(依頼の型)で実践に落とし、第6回のまとめ「実現力の上限は上がり、方向性を決める仕事はむしろ重くなった」で回収する。

## 2. 想定読者

- AIを仕事に使い始めている一般のビジネスパーソン・個人事業主。エンジニアに限定しない。
- ChatGPT や Claude を「検索の代わり」「文章の下書き」程度には使ったことがある。
- API・トークン・コンテキストウィンドウなどの用語は**知らない前提**。初出時に一文で説明する。
- 有料プラン(Pro等)の加入を検討中、または加入済み。

### 読者が連載を読み終えたときの状態(ゴール)

1. Fable 5 が従来のAIと何が違うかを、自分の言葉で説明できる
2. 目の前の仕事を「Sonnet/Opus で十分」「Fable 5 の出番」に振り分けられる
3. Fable 5 に大きな仕事を任せるときの、依頼文(プロンプト)の組み立て方が分かる
4. コスト・利用上限・安全機構(分類器)という3つの制約を理解して使える

## 3. 連載構成(全6回+展望コラム)

### 第1回: Fable 5 とは何か、何が得意か

- **目的:** 「賢いチャットAI」ではなく「長い仕事を任せられるAI」という認識の転換。
- **骨子:**
  1. 一言でいうと — 長く複雑な仕事を、途中で文脈を失わずに進めるAI
  2. 位置づけ — Opusクラスの上に新設された「Mythosクラス」初の一般提供。Mythos 5 との関係は1段落で軽く(同じ基盤モデル、違いは安全対策の有無)
  3. 得意なこと3つ — (a) 大量資料の読解(100万トークンコンテキスト) (b) コーディング(SWE-bench Pro 80.3% vs Opus 4.8 の69.2%、数字は最小限に) (c) 複数工程の自律作業(計画→検証→自己修正。Stripeの事例)
  4. 得意"でない"こと — 2つの軸で示す。(a) コスト面: 短い質問・日常タスクは不合理(第2回への橋渡し) (b) 質の面: ゼロからの独創(デザインの美的発想、記事の切り口、設計方針)は平均的な答えに収束する。ここで裏テーマ「方向性の決定は人間、実現はAI」を初出させる
  5. 知っておくべき現状 — 提供再開直後であること、料金体系、分類器による Opus 4.8 への迂回
- **次回への引き:** 「では、いくらかかるのか。どう使い分けるのか」

### 第2回: コストの仕組みとモデルの使い分け

- **目的:** 読者が自分のタスクを Sonnet / Opus / Fable 5 に振り分けられるようになる。
- **骨子:**
  1. トークンとは何か(初学者向けに一度だけ丁寧に)
  2. 3モデルの価格比較と、体感的な例え(「1回の依頼でいくら」の目安)
  3. 振り分けの判断基準 — 「失敗コストが高い仕事」「やり直し指示が何度も必要な仕事」ほど Fable 5 が割に合う
  4. 判断フローチャート(図を1枚)
  5. 実例: この連載自体が「設計は Fable 5、執筆は Sonnet 5」で作られている(制作の舞台裏)
- **依存:** 第1回の「得意/不得意」を前提にする。

### 第3回: 大量の資料を読ませる — 100万トークンの活かし方

- **目的:** 最も再現しやすい高価値ユースケースを1つ深掘り。
- **骨子:**
  1. 100万トークン = 何ページ分か(体感できる換算)
  2. 具体例: 契約書一式のレビュー、社内規程の横断チェック、長い議事録群からの論点抽出
  3. 依頼文の実例(Before/After)
  4. 注意: 読ませた内容の事実確認は人間の仕事
- **依存:** 第2回のコスト感覚(大量入力=入力トークン費用)。

### 第4回: 長い仕事を任せる — 複数工程タスクの頼み方

- **目的:** 「1問1答」から「仕事の丸ごと依頼」への発想転換と、その依頼文の型。
- **骨子:**
  1. 従来: 人間が細かく指示 → Fable 5: 目的を渡すと工程を分解して進める
  2. 良い依頼の3要素 — 目的・制約・完成条件
  3. 実例1本を通しで見せる(例: 「競合5社の調査→比較表→示唆レポート」)
  4. 途中経過の確認ポイント(丸投げしっぱなしにしない)
- **依存:** 第1回の「自律作業が得意」。

### 第5回: コーディングでの使いどころ

- **目的:** 非エンジニア読者には「開発をどう依頼するか」、エンジニア読者には「どの規模から割に合うか」。
- **骨子:**
  1. 単発のスクリプト・小修正 → Sonnet で十分
  2. Fable 5 の出番 — 複数ファイルにまたがる変更、大規模移行、テスト込みの品質(FrontierCode の「マージできる品質」の話)
  3. Claude Code との組み合わせに軽く言及(深入りしない。別連載候補)
- **依存:** 第2回の振り分け基準。

### 第6回: 制約と注意点 — 安全分類器・利用上限・データ保持

- **目的:** トラブル前に知っておくべきことをまとめ、連載を締める。
- **骨子:**
  1. 安全分類器 — サイバーセキュリティ等の依頼はブロックされ Opus 4.8 が代わりに応答する。無害な依頼が誤検知される可能性もある
  2. 利用上限と料金体系(執筆時点の最新を必ず再確認)
  3. データ保持 — Fable 5 は全通信を原則30日間保持(訓練には未使用)。機密情報を扱う組織は事前にポリシー確認を
  4. 総まとめ: 「その仕事は Fable 5 に頼む価値があるか?」チェックリスト。裏テーマを回収して締める — 「実現力の上限は上がった。方向性を決める仕事は、むしろ重くなった」
- **依存:** 全回。連載の総括。

### 展望コラム(番外編): AIは一つに収束しない — 垂直特化モデルは生き残るか

- **目的:** 連載の主題「使い分け」を業界構造の視点に拡張し、読者が今後のモデル選びの「地図」を持てるようにする。
- **位置づけ:** 本編6回の完結後に掲載する考察記事。他の回と異なり「筆者(とAI)の見立て」を含むため、**予想は予想と明記**し、断定調を避ける。事実パートと見立てパートを見出しで分ける。
- **骨子:**
  1. 性能競争の裏で進む多様化 — 3つの軸で整理: (a) 価格と性能の階層(Haiku〜Mythosクラス、Fable 5とSonnet 5の同時期展開) (b) 製品形態(チャット / Claude Code / Cowork / Claude Design と、モデル層より上のプロダクト層で分化が速い) (c) 規制と提供範囲(Fable 5 / Mythos 5 の分化、輸出規制の経緯)
  2. 逆向きの力 — 汎用性能が上がると専門特化モデルを追い抜いてしまう現象。特化モデルは「来年の汎用モデル」と競う構造的不利
  3. 見立て(非対称な予想) — 「言語で完結する垂直特化(医療文書・法務など)」は独立モデルとしては残りにくい。「言語の外にある特化(科学計算・構造予測など)」と「小さく・速く・安くあるべき用途(端末上推論・不正検知など)」は残る
  4. 価値の移動 — 生き残る垂直AI企業の堀は、モデルの重みではなく「非公開データへのアクセス権」と「業務ワークフローへの埋め込みの深さ」へ移る
  5. 予想を崩しうる条件 — (a) 規制が監査可能な独立モデルを制度的に要求する場合 (b) 外に出ない非公開データが性能を決める領域 (c) この見立て自体が汎用モデル(AI)の自己申告を含むというバイアスの開示
  6. 読者への含意 — 「一つのAIに忠誠を誓う」のではなく目的ごとに道具を選ぶ。個人開発・中小の実務では独自モデルは不要で、価値は自分のデータとワークフローの側にある。モデル呼び出しは差し替え可能な部品として設計する
- **執筆メモ:** 骨子2〜3で具体的な固有名詞事例(汎用に抜かれた特化モデルの例、科学分野の専用モデル名など)を挙げる場合は、ファクトシートにないため必ず【要確認】マークを付け、人間が出典を確定してから記載する。
- **依存:** 全回。特に第2回(使い分け)の視点を業界レベルに引き上げる関係。



- 用語の丁寧な初出説明は1回だけ。2回目以降は「(第◯回参照)」で流す。
- 各回は単独でも読めるようにするが、前提説明は2〜3行に留めてリンクで済ます。
- 事例の重複禁止。各回の事例は設計書のこの節で先取りして予約済みとする。

## 4. 執筆ガイド(Sonnet 5 への指示の基礎)

### 文体・トーン

- です・ます調。断定は根拠がある箇所のみ。煽り表現(「衝撃」「最強」「必見」)は使わない。
- 読者を「あなた」と呼ぶ。見下ろさない。「〜しましょう」より「〜すると楽です」のような提案形を基本にする。
- 1段落は3〜4文まで。1文は60字前後を目安に。
- 比喩は1記事に1〜2個まで。AIを人格化しすぎない(「Fable 5 が考えてくれる」はOK、「Fable 5 くんが頑張る」はNG)。

### 分量・構造

- 1記事 3,000〜4,000字(見出し・図表キャプション除く)。
- 見出しは H2 を4〜6個、H3 は必要な場合のみ。箇条書きは1記事3ブロックまで(箇条書きだらけにしない)。
- 図表は1記事1〜2点。第2回のフローチャートは必須。

### 記事の型(全回共通)

1. **冒頭(3〜5行):** 読者の具体的な状況・疑問から入る。連載の紹介はしない(初回のみ1行)。
2. **本文:** 骨子に沿う。実例は必ず「依頼文(プロンプト)の実物」を1つ以上含める。
3. **締めの定型:** 「その仕事は、Fable 5 に頼む価値があるか?」に対する本回の答えを2〜3行でまとめる。
4. **次回予告:** 1行。
5. **注記(全記事末尾に固定):** 「本記事の情報は◯年◯月時点のものです。料金・提供状況は変わる可能性があるため、最新は Anthropic 公式情報をご確認ください。」

### 用語統一表

| 使う | 使わない | 備考 |
|---|---|---|
| Fable 5 | フェイブル5、Claude Fable 5(2回目以降) | 初出のみ「Claude Fable 5(フェイブル・ファイブ)」 |
| コンテキスト(読み込める量) | 文脈長、コンテキストウィンドウ単独 | 初出時に「一度に読み込める量」と説明 |
| 依頼文 | プロンプト単独 | 初出時に「依頼文(プロンプト)」と併記、以後「依頼文」 |
| 安全分類器 | セーフガード、クラシファイア | |
| 従量課金 | クレジット制 | 初出時に補足可 |
| 使い分け | 出し分け、振り分け(見出しでは) | 本文中の動詞としての「振り分ける」は可 |

### 禁止事項

- 出典のない数値・事例を書かない。数値は本設計書のファクトシートにあるものだけを使い、それ以外が必要なら人間に確認を求める。
- 料金・提供状況・利用上限は**執筆のたびに要再確認**とマークする(ファクトシート参照)。
- 他社モデル(GPT、Gemini等)の批判・優劣断定をしない。比較はベンチマーク数値の紹介に留める。
- 「必ず」「絶対」「100%」を成果の文脈で使わない。

## 5. ファクトシート(2026年7月2日時点・確認済み)

> **重要:** ★印は変動が確実な項目。各回の執筆時に必ず一次情報(Anthropic 公式)で再確認すること。

### モデルの位置づけ

- 2026年6月9日に Anthropic が発表・提供開始。Opusクラスの上の新階層「Mythosクラス」初の一般提供モデル。
- Mythos 5 と同一の基盤モデル。違いは安全分類器の有無。Mythos 5 は Project Glasswing 経由の限定提供。
- 名前の由来: ラテン語 fabula(語られるもの)。ギリシャ語 mythos と通じる。

### スペック・料金

- コンテキスト: 100万トークン / 最大出力: 12.8万トークン。テキストと画像を入力可能。
- API料金: 入力 $10 / 出力 $50(100万トークンあたり)。Opus 4.8 の2倍、Sonnet 4.6 の3倍以上。
- ★サブスクプラン(Pro/Max/Team/Enterprise プレミアムシート)では2026年7月7日まで週間利用上限の最大50%まで利用可。7月8日以降は従量課金(クレジット)に移行予定。

### 性能(公式発表ベンチマーク)

- SWE-bench Pro: 80.3%(Opus 4.8: 69.2%、GPT-5.5: 58.6%)
- FrontierCode Diamond: 29.3%(Opus 4.8: 13.4% の2倍以上)
- 導入事例: Stripe「数ヶ月分のエンジニアリング作業を数日に圧縮」。5,000万行規模の Ruby コード移行を約2ヶ月見積→大幅短縮の事例。
- 注意: 生物学・サイバー系ベンチマークは分類器の影響を受けるため通常利用の体感とは分けて読む。

### 安全機構・制約

- 安全分類器がサイバーセキュリティ・生物・化学等の依頼を検知するとブロックし、Opus 4.8 が代わりに応答(ユーザーに通知される)。無害な依頼の誤検知(偽陽性)もありうる。
- ★2026年7月の再開版は分類器が強化され、コーディング・デバッグ等の一部の日常的タスクでも Opus 4.8 への切り替えが起きる場合があると Anthropic が告知。
- データ保持: Fable 5 / Mythos 5 の全通信は原則30日間保持(ゼロデータ保持の対象外)。訓練には使用しないと説明。企業利用ではコンプライアンス確認が必要。

### 経緯(第1回・第6回で言及)

- 6月9日公開 → 6月12日、米政府の輸出管理指令により提供全停止 → 6月30日、米商務省が規制解除を通知 → 7月1日、Claude Platform / Claude.ai / Claude Code / Claude Cowork で全世界に提供再開。
- 停止の発端: Amazon の研究者がセーフガード回避手法を報告。Anthropic は再開にあたり改良した分類器を導入(報告された手法を99%以上ブロック)。
