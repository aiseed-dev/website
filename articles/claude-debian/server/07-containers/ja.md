## なぜサーバーでコンテナを使うか

第6章では、aptでソフトを入れて、systemdのサービスとして動かす道筋をたどった。ファイル共有を一つ、メモ同期を一つ動かすぐらいなら、この方式で十分に回る。素直で、透明で、Debianの作法に忠実だ。

だが、動かしたいものが増えると、この方式は軋み始める。

**一つ目：依存の衝突。** アプリAは古いライブラリを、アプリBは新しいライブラリを要求する。同じ機械に両方をaptで入れると、片方が動かなくなることがある。一台のサーバーに複数のアプリを同居させた瞬間に、この衝突が現実になる。

**二つ目：消した跡の残骸。** `apt remove` でアプリを消しても、設定ファイル、ログ、作られたユーザー、データベースの断片が、あちこちに残る。何度か入れては消すを繰り返すと、サーバーは少しずつ「説明のつかない汚れ」を溜めていく。

**三つ目：バージョン違い。** Debianのパッケージは安定を重視するため、最新版でないことが多い。新しい機能が欲しいアプリでは、これが足かせになる。

コンテナは、この三つを一気に解く。**コンテナとは、アプリとその依存をまるごと一つの箱に閉じ込めて、ホストの環境から切り離す仕組みだ。** 箱の中は箱の中で完結し、外を汚さない。気に入らなければ箱ごと捨てて、作り直せる。

そして決定的なのは、その箱の設計図が `compose.yaml` という**一枚のテキストファイル**になることだ。何をどう動かすか、どのポートを開けるか、どこにデータを置くかが、すべて一枚の読めるテキストに書かれる。**テキストになった瞬間、それはClaudeの土俵だ。** この本が一貫して言ってきたこと——「テキストになるものはClaudeと一緒に扱える」——が、サーバーで最も強く効く場所が、ここだ。

## 第一節 DockerかPodmanか

コンテナを動かすツールは、Debian 13（trixie）では二つとも標準のaptで入る。DockerとPodmanだ。どちらを選ぶかで最初に迷うので、違いを正直に短く書く。

**Docker** は事実上の業界標準だ。世の中のチュートリアル、公式イメージの説明、トラブル時の検索結果——情報量が圧倒的に多い。困ったときに答えにたどり着きやすい、というのは初学者にとって大きな利点だ。一方で、常駐するデーモン（背後で動き続けるプロセス）がroot権限で動く、という設計上の重さがある。

**Podman** はデーモンを持たない設計で、一般ユーザーの権限のまま（rootlessで）コンテナを動かすのが素直だ。セキュリティ的にはこちらの方が筋がいい。コマンドはDockerとほぼ互換で、`docker` を `podman` に置き換えるだけで多くが動く。ただし情報量はDockerに一歩譲る。

本章はDockerで進める。理由は情報量の多さだ。だが、**本章の内容はPodmanでもそのまま成立する。** compose.yaml の書き方も、データとコンテナを分ける考え方も、判断基準も、ツールに依存しない。後でPodmanに乗り換えたくなっても、ここで学んだことは無駄にならない。

```bash
# Docker本体とcomposeプラグインを入れる
sudo apt update
sudo apt install docker.io docker-compose-v2

# 動いているか確認
sudo systemctl status docker
docker --version
docker compose version
```

`docker.io` がDocker本体、`docker-compose-v2` が `docker compose` という複数コンテナをまとめて扱うサブコマンドだ。古い記事には `docker-compose`（ハイフン付き、独立コマンド）が出てくるが、今は `docker compose`（スペース区切り、プラグイン）が標準だ。

毎回 `sudo` を付けたくなければ、自分のユーザーをdockerグループに入れる。

```bash
# 自分をdockerグループに追加（ユーザー名は自分のものに）
sudo usermod -aG docker $USER
# 反映には再ログイン（またはサーバーごと再起動）が必要
```

ここは正直に書いておく。**dockerグループに入るということは、実質的にroot権限を手に入れるということだ。** Dockerのデーモンはroot権限で動いているので、それを操作できる人間は、コンテナ経由でホストのファイルを好きにできてしまう。一人で使う実験機なら気にしなくていいが、複数人が入るサーバーでは、誰をdockerグループに入れるかは慎重に判断する。この「実質root」の意味は、第5章で扱った脅威モデルの延長線上にある。

### Claudeに聞いてみよう①：DockerとPodmanのどちらが自分向きか

> 私はDebian 13で自宅サーバーを運用しています。状況は次のとおりです：
> 〔my-server.md を貼る。特に「利用者は自分だけか/複数か」「LAN内だけか/公開予定か」「動かしたいアプリ」を明記〕
>
> この状況で、コンテナ実行環境はDockerとPodmanのどちらが向いていますか。セキュリティ・情報量・乗り換えやすさの観点で、私の状況に即して理由を添えて教えてください。

ツール選びは思想の話になりがちだが、自分の状況を渡せばClaudeは具体的な天秤にかけてくれる。一人の実験機ならDocker、人が増えるならPodmanを検討、といった当たり前の結論でも、自分の状況で確認しておくと迷いが消える。

## 第二節 最初のコンテナと compose

### まずは一個動かして疎通を確認する

理屈の前に、一個動かしてみる。Dockerには疎通確認専用の小さなイメージがある。

```bash
# 動作確認用の最小コンテナ。実行後に自動で消える(--rm)
docker run --rm hello-world
```

「Hello from Docker!」と表示されれば、イメージのダウンロードからコンテナの起動・終了まで、ひと通りの経路が通っている。`--rm` を付けたので、終わったコンテナはゴミを残さず消える。これがコンテナの気楽さの第一歩だ。

### compose.yaml という設計図

`docker run` を毎回長い呪文として打つ代わりに、構成を一枚のテキストに書き出して、それを実行する。それが `compose.yaml` だ。例として、Webサーバーの nginx を 8080 番ポートで立ち上げ、自分で用意したHTMLを差し込む構成を、丸ごと載せる。

まず作業用のディレクトリと中身を用意する。

```bash
mkdir -p ~/webtest/html
echo '<h1>私の最初のコンテナ</h1>' > ~/webtest/html/index.html
```

その `~/webtest/` の中に、`compose.yaml` を次の内容で作る。

```yaml
services:
  web:
    image: nginx:stable          # 使う公式イメージ
    ports:
      - "8080:80"                # ホストの8080をコンテナの80につなぐ
    volumes:
      - ./html:/usr/share/nginx/html:ro  # 手元のhtmlを読み取り専用で差し込む
    restart: unless-stopped       # 落ちても自動で起き直す
```

このディレクトリで次を打てば、構成どおりにコンテナが立ち上がる。

```bash
cd ~/webtest
docker compose up -d        # -d は背後で動かす(デタッチ)

docker compose ps           # 動いているコンテナの一覧
docker compose logs -f      # ログを流しっぱなしで見る(Ctrl+Cで離脱)
```

ブラウザや別端末から `http://サーバーのアドレス:8080` を開けば、さっき書いたHTMLが出る。止めて片付けるのは一行だ。

```bash
docker compose down         # コンテナを止めて消す
```

`down` でコンテナは消えるが、`./html` に置いたファイルは手元にそのまま残る。ここが肝だ。

### データはボリュームに、コンテナは使い捨てに

コンテナの設計でいちばん大事な原則を、一つだけ覚えてほしい。**消えて困るデータは、コンテナの外（ボリュームかバインドマウント）に置き、コンテナ自体は使い捨てにする。**

- **バインドマウント**：上の例の `./html:/usr/share/nginx/html` のように、ホストのディレクトリをコンテナの中に直結する方式。中身が手元のファイルとして見えるので、設定ファイルや小さなデータに向く。
- **ボリューム**：Dockerが管理する保管領域を、コンテナの中に差し込む方式。データベースのような大きく動きの激しいデータに向く。

どちらを使っても、原則は同じだ。**データはコンテナの寿命と切り離す。** こうしておくと、アプリのバージョンを上げたいときは「コンテナを捨てて新しいイメージで作り直す」だけで済む。データは外に残っているので無傷だ。この「コンテナは使い捨て、データは外」という分離が、第9章で扱うバックアップの大前提になる。守るべきデータの置き場所がはっきりしていれば、バックアップは劇的に簡単になる。

### Claudeに聞いてみよう②：動かしたいアプリの compose.yaml を書かせる

> 私はDebian 13のサーバー（Docker導入済み）で、〔アプリ名。例：Nextcloud / Vaultwarden / Gitea / 自作のWebアプリ など〕を動かしたいです。サーバーの状況は次のとおりです：
> 〔my-server.md を貼る。メモリ・ディスク・LAN内かどうかなど〕
>
> このアプリの compose.yaml を書いてください。そのうえで、**各行が何をしているのかを一行ずつ解説**してください。データを失わないために、どのディレクトリをボリュームかバインドマウントにすべきかも明示してください。

ここがこの章の核心だ。compose.yaml を「書かせる」だけでなく、**各行の意味を説明させる**ところまでをセットにする。意味の分からない設定を貼り付けて動かすのは、ブラックボックスを一つ増やすのと同じだ。一行ずつ解説させれば、次に似た構成を自分で読めるようになる。

## 第三節 コンテナにしない方がいいもの

ここまで読むと「全部コンテナにすればいい」と思うかもしれない。だが、それは「全部aptで入れる」のと同じくらい極端だ。コンテナに向くものと、向かないものがある。

向かないものの代表は、**ホストの根幹に関わるものと、ハードウェアに近いもの**だ。

- **sshd（SSHの待ち受け）**：サーバーに入るための入口そのもの。これをコンテナに入れると、コンテナが壊れたときにサーバーに入れなくなる本末転倒が起きる。ホストで直接動かす。
- **ファイアウォール（ufw / nftables）**：ホスト全体を守る仕組みなので、ホストで動かすのが筋。第5章で設定したものをそのまま使う。
- **時刻同期・カーネル・ドライバ**：ハードウェアやOSの土台に近いものは、コンテナの外、ホストの責任で持つ。

逆に、コンテナに向くかどうかは、次の三つで判断するといい。

1. **状態を持つか。** Webアプリやサービスのように「動かして、いつか作り直す」性質のものはコンテナ向き。逆に、ホストの設定そのもののように「一度決めたら動かさない」ものは、ホストに置く方が素直だ。
2. **頻繁に入れ替えるか。** バージョンを試したり、入れては消したりするものは、コンテナの使い捨てやすさが効く。
3. **公式イメージの質。** 開発元が公式に配っているイメージがあり、保守されているなら安心して使える。素性の知れないイメージは、ブラックボックスを抱え込むことになるので避ける。

**「全部コンテナ」も「全部apt」も極端だ。** ホストの根幹はホストで素直に、入れ替えの激しいアプリはコンテナで気楽に——この使い分けが、壊れにくく、説明のつくサーバーを作る。

### Claudeに聞いてみよう③：既存の compose.yaml をセキュリティレビューさせる

> 私が今動かしている、または使おうとしている compose.yaml です：
> ```
> 〔compose.yaml の中身をそのまま貼る〕
> ```
>
> この構成を、セキュリティの観点でレビューしてください。特に次を見てほしいです：不要に外へ開いているポート、平文で書かれた危険なパスワードや鍵、`privileged` などの過大な権限、ボリュームの危険なマウント。各指摘について、なぜ危険かと、推奨する直し方を添えてください。

ネットで拾った compose.yaml をそのまま動かす前に、この一往復を挟む癖をつけたい。便利な設定例ほど「とりあえず全権限・全ポート開放」になっていることがある。第8章で外に公開する前に、この習慣が効いてくる。

## 第四節 Claudeとコンテナ運用

コンテナ運用の日常は、突き詰めると三つの動きの繰り返しだ。**書かせる、説明させる、自分の状況に直させる。**

- **書かせる**：動かしたいアプリ名を伝えて、compose.yaml の叩き台を作らせる(①)。
- **説明させる**：出てきた各行の意味を一行ずつ解説させる。意味の分からない行を残さない。
- **直させる**：ポート番号、ボリュームの場所、環境変数（パスワードやドメイン名など）を、自分のサーバーの状況に合わせて書き換えさせる。

そして、何かが動かないときの最強の一手が、ログをそのまま貼ることだ。コンテナが起動直後に落ちる、ポートが開かない、データベースにつながらない——その原因はほぼすべて、ログに書かれている。

```bash
# 落ちたコンテナを含めてログを見る
docker compose logs

# 特定のサービスだけ、直近を追う
docker compose logs --tail=50 web
```

この出力をそのままClaudeに貼れば、エラーの意味と直し方が返ってくる。第6章で `journalctl` のログを貼ったのとまったく同じ作法だ。**サーバーで起きることはすべてテキストになり、テキストはClaudeと一緒に読める。** コンテナでも、この原則は変わらない。

### Claudeに聞いてみよう④：compose のエラーログを診断させる

> 私の compose.yaml は次のとおりです：
> ```
> 〔compose.yaml を貼る〕
> ```
>
> `docker compose up -d` のあと、コンテナが正常に動きません。`docker compose logs` の出力はこれです：
> ```
> 〔ログをそのまま貼る〕
> ```
>
> 何が起きているか、原因と直し方を教えてください。修正後の compose.yaml と、確認のために打つべきコマンドも添えてください。

エラーログを「読める形のまま」渡せるのが、サーバーとClaudeの相性の良さだ。スクリーンショットではなく、テキストでそのまま貼る——第3章で身につけた作法を、コンテナでもそのまま使えばいい。

## まとめ

この章でやったこと：

1. aptで直接入れる方式の限界（依存の衝突・残骸・バージョン違い）を確認し、コンテナがそれを解く理由を理解した
2. DockerとPodmanの違いを把握し、Dockerを入れて疎通確認した（dockerグループ＝実質rootの意味も含めて）
3. compose.yaml の最小実例（nginxを8080で公開）を書いて、起動・確認・停止のひと回りをやった
4. 「データは外、コンテナは使い捨て」の原則と、コンテナに向く/向かないものの判断基準を整理した
5. compose.yaml をClaudeに書かせ・説明させ・直させ、エラーログを診断させる型を身につけた

手元に残ったもの：
- 動いた compose.yaml（最小のnginx）一枚と、それを動かすコマンドの記憶
- 動かしたいアプリの compose.yaml の叩き台（①②で作ったもの）
- 「コンテナに向く/向かない」の判断基準のメモ

次の第8章では、ここまでLAN内で安全に練習してきたサーバーを、いよいよ**外の世界に公開する**ことを考える。ドメイン、リバースプロキシ、TLS証明書——公開は不可逆に近い決断だ。「公開しない」という立派な選択肢も含めて、自分の回線と状況に合った公開経路を、Claudeと一緒に設計する。
