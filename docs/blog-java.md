### 2. AI時代の開発の実態(500字程度)

人間が書くのは Markdown(構造・仕様)で、実装は AI が担当する。データを扱うのに、もうクラス階層を設計する必要がない。スキーマを自然言語で書けば、AI が SQL と Python コードを生成する。

ただし、Markdown でスキーマを書くには、業務知識(具体)と構造化能力(抽象)の両方が必要である。この両方を持つ人は、思ったほど多くない。

### 3. 実例1:顧客管理

#### 3.1 Markdown仕様

```markdown
## 顧客データ

顧客は以下を記録する:
- 顧客ID(自動採番、必須)
- 顧客名(必須)
- 顧客種別(必須、以下のいずれか:個人、法人)
- 郵便番号(任意)
- 住所(任意)
- 電話番号(任意)
- メールアドレス(任意)
- 登録日(自動)
- 最終更新日(自動)

ルール:
- メールアドレスは形式が正しいこと
- 法人の場合は会社名を顧客名に入れる
- 個人の場合は氏名を顧客名に入れる
```

上記のMarkdown仕様をAIに渡して生成されたSQLite + Pythonのコードを掲載する。生成方針:

- ファイル数は1
- ORMは使わない(生のSQL)
- 関数ベース(クラスは作らない)
- データの追加・取得・更新の関数を含める
- 行数は40〜60行程度

- CustomerRepository.cs(リポジトリパターン)

ファイル数と総行数を明示する。完全な動作コードでなくてよい(構造が伝わればよい)。

### 4. 実例2:受注管理

#### 4.1 Markdown仕様

```markdown
## 受注データ

受注は以下を記録する:
- 受注番号(自動採番、必須)
- 受注日(必須)
- 顧客ID(顧客テーブルへの参照、必須)
- 受注状態(必須、以下のいずれか:受付、確認中、確定、出荷、完了、キャンセル)
- 備考(任意)
- 登録日時(自動)
- 更新日時(自動)

## 受注明細データ

1つの受注に複数の明細を含む。明細は以下を記録する:
- 受注番号(受注テーブルへの参照、必須)
- 明細番号(受注内での連番、必須)
- 商品ID(商品テーブルへの参照、必須)
- 数量(必須、1以上の整数)
- 単価(必須、受注時点の値を保存)

ルール:
- 受注番号は重複しない
- 受注状態の遷移は決められた順序: 受付 → 確認中 → 確定 → 出荷 → 完了

- キャンセル後は他の状態に戻せない
```

### 6. コード量から見える16倍、実質100倍の差(1000字程度)

業務アプリの典型例3つで、Java/C# と Python の間にコード量で4〜10倍の差が出る。

この差は、新規開発のコストだけでなく、メンテナンスのコストにも掛け算で効いてくる。

#### 機械的な計算:16倍

AIにコードを修正させるとき、二つのコード量が関係する:

**書くコードの量(4倍)**
- 同じ機能を実装するのに、C#はPythonの約4倍のコードを書く必要がある
- AIが生成するトークン数が4倍
- 生成時間が4倍

**参照するコードの量(4倍)**
- 修正時に読む必要があるコードが、C#は約4倍ある
- ファイル分散により、関連ファイルを全部見ないといけない
- AIに入力するコンテキストが4倍

この二つは独立に効くのではなく、掛け算で効く:

**機械的な計算では、C#のメンテナンスはPythonの16倍のコストがかかる**(4倍 × 4倍)

#### 実質:約100倍

しかし、これはコード量だけの計算である。実際にAIに修正させる現場では、更にコストが膨らむ:

- **AIの精度の差**: Pythonは標準パターンが学習データに豊富にあるため、一発で正しく書ける。C#はカスタムクラスが多く、AIの試行回数が増える
- **修正の連鎖**: クラスを修正すると、インターフェース、実装、テスト、DI設定など複数箇所に波及する。一箇所修正したら別の箇所が壊れる
- **AIが正しく修正できない**: ファイル分散でAIが全体を把握できず、修正の整合性が取れないことがある
- **デバッグの困難さ**: EFのエラーメッセージは抽象的で、暗黙の動作が絡むため原因特定に時間がかかる
- **ビルド・実行サイクルの遅さ**: C#は試行錯誤に時間がかかる

これらが掛け合わさって、**実質的には約100倍の差**になる。

#### 何年も続くコスト

これは新規開発時の差ではない。**コードが書かれた後、何年も続くメンテナンス期間の差**である。

業務アプリは作って終わりではない。法令改正、業務変更、機能追加で、何度も修正が必要になる。そのたびに、100倍のコストが積み重なる。

加えて、3つの実例のMarkdown仕様は、いずれも業務に詳しく構造化できる人なら書ける内容である。これは偶然ではない。自然言語でスキーマを定義するアプローチの本質的な利点である。

### 7. 業務アプリを作るのは誰か(800字程度)

AI時代に、業務アプリを作るのは誰か。

従来の答え:**ソフトウェアエンジニア**
新しい答え:**業務に詳しく構造化できる人**

業務に詳しく構造化できる人とは、業務知識(具体)と構造化能力(抽象)の両方を持つ人である。具体的には:
- 個人事業主(自分の事業の仕組みを最もよく知っている)
- 地域の事業者(現場の運用を体で覚えている)
- 自由人(複数の領域に関わって、構造を捉えている)
- 業務担当者(構造化能力を持つ人に限る)

これらの人々は、従来は「エンジニアではない」という理由で、自分の業務のためのアプリを作ることから排除されてきた。プログラミング言語を知らない、フレームワークを知らない、設計パターンを知らない——だから自分で作ることはできなかった。

しかし、AI時代には、これらの「知らない」が問題にならない。必要なのは:
- 自分の業務を言葉にする力
- 何を記録し、どう使うかを構造として捉える力
- AIと対話する力

これらはプログラミングのスキルではない。現場を観察し、構造を捉え、本質を言葉にする力——リベラルアーツ的な能力である。

ただし、この能力を持つ人は、実は多くない。業務を知っているだけでは足りない。構造化できるだけでも足りない。**両方持つ人**が、AI時代の業務アプリの担い手になる。これは中世の自由人と似た立場である。

中世の自由人(liber homo)は、土地に縛られず、ギルドに属さず、教会の階層にも従属せず、複数の領域を横断し、自分で判断して生きる人々だった。読み書きができ、複数の知識を持ち、構造を見抜けた。リベラルアーツ(自由学芸)は、自由人にふさわしい教養として発展した。

AI時代の自由人も、構造は同じである。組織や専門性に縛られず、複数領域を横断し、自分で判断する。AIと対話できる。構造を読める。

自然言語でスキーマを書けば、AIが確実にコード化する。この事実が、業務アプリ開発を、エンジニアの専門領域から、業務に詳しく構造化できる自由人の領域へと広げている。

ただし、これは「誰にでも開かれている」わけではない。**業務知識と構造化能力の両方を持つ人にとっては**、開かれた道がある。

### 8. 想定される反論への応答(各200字程度)

#### 「Pythonは動的型付けだから業務アプリには向かない」

型ヒント、mypy、pydanticで実用上の型安全性は確保できる。Java/C#の重い型システムは、AIに書かせる時代には不要なオーバーヘッドになっている。

#### 「大規模システムにはJava/C#が必要」

「大規模システム」という前提自体が時代遅れになりつつある。AI時代には、個人または小規模チームで業務アプリを開発できる。

#### 「業務に詳しいだけの人が作ったアプリは品質が低いのではないか」

業務に詳しく構造化できる人が作ったアプリは、業務を理解していない専門エンジニアが作ったアプリより、しばしば実用的である。AIが書くコード自体の品質は、標準ライブラリと有名ライブラリの組み合わせで担保される。

#### 「セキュリティや法令対応はどうするか」

セキュリティの基本は、AIが書くコードに標準的に組み込まれる。法令対応は、業務に詳しい人の方が、専門エンジニアより詳しいことも多い。

#### 「既存のJava/C#資産はどうする」

既存資産を捨てる必要はない。だが、新規開発でJava/C#を選ぶ理由はもうない。そして、既存のJava/C#資産は、AI時代のメンテナンスコストがPythonの100倍かかる。長期的には、段階的なPythonへの移行を検討する価値がある。

#### 「AIが書くコードは本当に信頼できるのか」

Anthropic、OpenAI、Googleといった最先端のAI企業ですら、AIモデル自体のコードをAIに書かせている。AIが世界最高峰のAIモデルのコードを書ける時代である。業務アプリのコードを書けないはずがない。

### 9. 結論(500字程度)

AI時代の業務アプリ開発において、Java と C# の時代は終わった。

ソフトウェアエンジニアリングは、AIが完全に身につけた。最先端のAI企業ですら、AIモデル自体のコードをAIに書かせている。コードを書くことは、もう人間の仕事ではない。

業務アプリ開発における二つの転換が起きている:
- **言語**: Java/C# から Python へ
- **担い手**: 専門エンジニアから、業務に詳しく構造化できる人(自由人)へ

両方の転換の根拠は、一つの事実にある。**自然言語のMarkdownスキーマがあれば、AIは確実にコード化できる**。

そして、**AIによるメンテナンスコストは、機械的な計算でC#がPythonの16倍、実質的には約100倍**かかる。書くコードが4倍、参照するコードが4倍、掛け合わせて16倍。これにAIの精度、修正の連鎖、デバッグの困難さが乗って、約100倍になる。

業務アプリは作って終わりではない。何年もメンテナンスが続く。そのたびに、100倍のコストが積み重なる。

「業務アプリ = Java/C#」「業務アプリ = ソフトウェアエンジニアの仕事」という二つの常識は、2000年代から2010年代に固まった社会的な慣性にすぎない。技術的な根拠はもう薄れている。

アプリを作る仕事自体は、なくならない。むしろ、今まで作れなかったものが作れるようになる分、広がっていく。担い手が変わるだけである。業務を知り構造化できる人にとっては、開かれた道がある。これは中世の自由人と似た立場である。組織や専門性に縛られず、複数領域を横断し、自分で判断できる人なら、この道に入っていける。

- 特定の企業を批判しない(Microsoft、Oracleなど)
- 神経多様性に過度に焦点を当てない(自由人の例の一つとして自然に含める)
- 「誰でも作れる」と楽観的すぎる表現を使わない(業務知識と構造化能力の両方が必要)

## 推奨する書き方

- 各セクションは短く区切る
- 一つの段落は3〜5行程度
- 実例のMarkdown仕様とPythonコードは、コードブロックでそのまま掲載
- 比較は表形式で見せる
- 結論は短く、潔く
- 「業務に詳しく構造化できる人」「自由人」「現場の人」という言葉を自然に使う
- 「16倍」「100倍」という具体的な数字を、結論部とセクション6で明示的に出す
- 「終焉」というタイトルにふさわしい、断言する文体を維持する
- 「ソフトウェアエンジニアリングは、AIが完全に身につけた」を冒頭と結論で明示する
- 中世の自由人との対比を、セクション7と結論で軽く触れる
- 数字(16倍、100倍)は明確に出す
- AnthropicがClaudeのコードをClaudeに書かせている事実を、根拠として活用する
