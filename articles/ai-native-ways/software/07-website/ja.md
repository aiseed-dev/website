# Webサイトを作る ── AI と対話して

第5章で見たとおり、顧客は AI と組んで自分で作れる。その実例の一つが、
会社サイトやブログ ── Web サイトだ。その大半は **静的**でよく、静的なら、
自分のサーバすら要らない。

## WordPress はサイバー攻撃に弱い ── まず、ここから直す

世界の Web サイトの多くは WordPress で動いている。便利な反面、**サイバー
攻撃の最大の標的**でもある。理由は、その構造にある。

- **動くサーバがある** ── PHP がリクエストごとにコードを実行し、データ
  ベースに問い合わせる。動く部分が多いほど、攻撃される面は広い。
- **プラグインとテーマ** ── 機能を足すために第三者製のプラグインを大量に
  入れる。更新の止まった・脆弱なプラグインが、侵入の最大の入口になる。
- **管理ログインがある** ── 管理画面のパスワードは、世界中から自動で総当たり
  され続ける。
- **数が多い** ── 最も使われている CMS だから、攻撃ツールは常に WordPress を
  狙って自動で走査している。古いまま放置されたサイトが、次々に乗っ取られる。

**とくに危ないのが、ネットショップ(e-shop)だ**。決済と顧客の個人情報を
扱うため、価値の高い標的になる。カード情報を盗むスキミング、個人情報の流出
── 被害が直接、お金と信用に響く。**だから、WordPress のサイト、中でも
e-shop は、まっ先に改善すべきだ**。

直し方は、シンプルだ。**静的サイトにすれば、これらの攻撃面はほぼ消える**
── 動く PHP もデータベースも管理ログインもプラグインも無い。攻撃する場所が
無くなる(ネットショップの決済は、Stripe などの外部の決済ページに任せ、
カード情報を自分で持たない ── これも攻撃面を減らす)。

> WordPress が攻撃されやすいのは、**動くサーバ・プラグイン・管理ログイン**が
> あるからだ。静的にすれば、その面ごと消える。e-shop は最優先で。

## なぜ静的にするのか

社外サイトの中身は、めったに変わらない。だから毎回サーバで組み立てず、
先に HTML を作って置く。そのほうが、速く、安全で、安い。

- **速い** ── ただのファイル配信。データベースを待たない
- **安全** ── 動く部分がないので、攻撃される面がほぼ無い(前節のとおり)
- **楽** ── 中身は Markdown。Claude に書かせ、ビルドが HTML にする

このサイト(aiseed.dev)も静的だ。Markdown から HTML を作って置いている。
動的なサーバは無い。

## 静的サイトの作成

サイトの仕組み ── Markdown を HTML に変換するビルドと、見た目のテンプレート
── は、最初に一度だけ作る。プロのフロントエンドエンジニアであれば、Astro の
ようなフレームワークを使ってもいい。しかし、はじめての人でも、**Claude に
指示すれば Web サイトを作ってくれる**。

コツは、一度の指示で決めようとせず、**対話で進める**ことだ。とくに初めての
人ほど、これが効く ── 最初から仕様を書き切れなくても、やり取りの中で
「こうしたい」が見えてくる。それに **Web は、後からいくらでも直せる**。作り
直して上げ直せばいいだけだから、完璧を狙わず、まず形にすればいい。まず素直に
切り出す ── 「Markdown で書いた記事を、HTML のサイトに変換するビルドを
作って」。あとは Claude とやり取りしながら詰めていく。話に出すとよいのは、
次のようなことだ。

- **何を書くか** ── 記事とブログを Markdown で。各ファイルの先頭にタイトル・日付・要約(frontmatter)を持たせる
- **URL の形** ── 例:記事は `/<slug>/`、ブログは `/blog/<slug>/`
- **一覧ページ** ── トップと各カテゴリに、新しい順で並ぶ索引
- **見た目** ── 文章中心で読みやすく、素の CSS で書く。凝った装飾や CSS フレームワーク(Tailwind 等)は使わない(既存サイトに似せたいなら、その URL や画像を渡す)
- **JavaScript** ── 原則、最小限のバニラ JS だけ。React のようなフレームワークは使わない(静的サイトに重い土台は要らない)
- **出力と画像** ── 変換結果は `html/` に出し、画像も一緒に運ぶ
- **多言語** ── 日本語と英語を併記するなら `/` と `/en/` に分ける
- **道具** ── ビルドは単一コマンドで動かす。npm や Node は使わない(Python など手元の物で)

このやり取りで、Claude が変換スクリプトとテンプレートを書く。気に入らなければ
「一覧を2列に」「日付の形式を変えて」と、出来た物を見ながら直す ── 対話で
形にしていく(第5章の作り方そのものだ)。一度できれば、あとは記事を書くだけ
になる。

## 既存サイトは、静的に吸い出す

すでにある WordPress サイトを一から作り直すのが大変なら、もっと速い手が
ある ── **今あるサイトを丸ごとたどって、HTML と画像を保存し、静的なファイル
の束にする**。動く WordPress を、そのままの見た目で、静的な置物に変える。

Mac には **SiteSucker** という定番ソフトがある(有料)。だが、同じことは
**Python でできる** ── しかも無料で、どの OS でも、中身の分かる形で。AI に
頼めば、`requests` でページを取り、`BeautifulSoup` でリンクと画像を拾って
保存し、リンクを相対パスに書き換えるスクリプトを、数十行で書いてくれる。

```python
# 概略: 既存サイトをたどって静的ファイルに保存する
import requests
from bs4 import BeautifulSoup
# 1) ページを取得 → html/ に保存
# 2) <a> <img> <link> <script> をたどり、同じドメインの物を保存
# 3) リンクを相対パスに書き換える
# (全体は AI と対話して仕上げる。コマンド一発なら wget --mirror でもいい)
```

吸い出した静的ファイルを Cloudflare Pages に上げれば、**動く WordPress を
止められる ── 攻撃面ごと**。問い合わせフォームや決済など「動く機能」は
残らないので、フォームは外部サービスへ、決済は Stripe などの外部ページへ
移す。

> 既存サイトは、**たどって静的に吸い出す**。SiteSucker(Mac・有料)と
> 同じことは、AI に頼めば Python で無料で作れる。

## 記事・ブログを書く

仕組みができたら、あとは **記事とブログを書くだけ**だ。これが日々の運用に
なる。中身は Markdown で書く。決まった置き場(例:`articles/`)に、記事を
一つファイルとして足し、ビルドで HTML に変換する。

```bash
# articles/ に Markdown を書き、ビルドで html/ にする
python tools/build.py
```

日々の運用は、**Markdown を書いて、ビルドして、上げる**、それだけ。出力は、
ただのファイル群(`html/`)。

## まとめ

- **WordPress はサイバー攻撃に弱い** ── 動くサーバ・プラグイン・管理ログインが標的。とくに e-shop は最優先で直す
- **静的にすれば攻撃面が消える** ── 動く PHP も DB も管理ログインも無い
- **仕組みは Claude と対話で一度だけ作る** ── Astro を学ぶ必要はない、初めてでも対話で
- **素の CSS・最小限のバニラ JS** ── 静的サイトに重い土台は要らない
- **運用は記事を書くだけ** ── Markdown を足してビルド、後からいくらでも直せる

ここまでが「サイトを作る」。**公開(Cloudflare Pages に上げる・ドメイン・
確認とデプロイ)** は、自立編で自分の側の道具と合わせて扱う。次章では、
もう一つの実例 ── **組み込み(機器を動かすソフト)** を AI と作る。
