# Microsoftにおけるトークン課金増大の嵐は、ツールの問題だ
## 文書をMarkdownで書き、コードをPythonで書こう

---

## 概要

2026年5月、Microsoftが自社の主力部門で Claude Code の利用停止を命じた。同時期、Uberは年間AI予算をわずか4ヶ月で使い切った。

エージェント型AIである Claude Code は、コードを深く読み込んで関連ファイルまで把握するからこそ高い能力を発揮する。だがこの「深く読む」性質は、扱うコードベースとファイル形式の構造に応じてトークン消費を爆発させる。

**特に Office + C#/.NET の組み合わせは、AI時代に構造的に最悪のコストを生む**:
- C#/.NETのファイル分散設計と抽象化レイヤー
- バリアント型を持たないことによるクラス階層の肥大化
- OOXML(.xlsx, .docx, .pptx)の重い構造
- これらが組み合わさることで生じる相乗的なトークン消費

Microsoftほどの企業ですらこのコストを正当化できなかった。同じ構造を持つ多くの日本企業が、近い将来同じ嵐に直面する。

一方、**個人ビルダーや小規模開発者はこの嵐を恐れる必要がない**。Markdown + Python + 標準フォーマット(JSON、SQLite、Parquet)で構築すれば、Claude MAX $100〜$200/月のプランで快適に開発を続けられる。これがAI時代における個人と小規模の構造的優位である。

## 1. Microsoftが自社でClaude Codeを止めた

2026年5月、Microsoftは「Experiences + Devices」部門(Windows、Microsoft 365、Outlook、Teams、Surfaceを担当)のエンジニアに、Claude Code の利用停止を命じた。期限は6月30日。代わりに自社製の GitHub Copilot CLI への移行が義務付けられた。

公式説明は「ツールチェーン統一」だったが、真の理由は財務的圧迫だった。社内で Claude Code が「人気になりすぎた」結果、Anthropicへのトークン課金が想定を大きく超えたのである。期限が会計年度末と一致していることが、この財務的動機を裏付けている。

同時期、Uberでも同じことが起きていた。CTOのPraveen Neppalli Naga氏は、2026年通期のAI予算が**わずか4ヶ月で枯渇した**ことを認めている。

| 時期 | UberのClaude Code利用状況 |
|---|---|
| 2025年12月 | 約5,000人のエンジニアに導入開始 |
| 2026年2月 | 利用率32% |
| 2026年3月 | 利用率84%、大半がエージェント的ユーザーに |
| 2026年4月 | 利用率95%、コミットコードの70%がAI由来 |
| 2026年4月末 | 年間予算が完全枯渇 |

Naga CTO自身が実施したわずか2時間のデモセッションで、$1,200のAPIコストが発生したという。ヘビーユーザーの月額消費は500〜2,000ドルに達した。

さらに象徴的な事例として、Anthropicは2026年4月、消費者向けプランで「OpenClaw」というオープンソースエージェントの実行を禁止した。月額$200の MAXプランで OpenClaw を自律稼働させた結果、1日あたり$1,000〜$5,000のAPIコストが裏で発生していたのである。

これらの事例は、AI業界が「定額制サブスクリプション経済」から「従量課金ユーティリティ経済」へ不可逆的に移行したことを示している。だが本稿が掘り下げたいのは、もう一段深い問いだ。

**Microsoftほどの資金力を持つ企業が、なぜ自社開発でAIを使い続けられないのか?**

答えは、Microsoft自身が作り上げたエコシステムの中にある。

## 2. エージェント型AIの能力とトークン爆発のメカニズム

Claude Code が高い能力を発揮するのは、それが**エージェント型AI**だからである。従来の行単位の補完ツールとは根本的に異なる。

Claude Code はコード修正を頼まれると、以下を自律的に行う:

1. リポジトリ全体の構造を把握する
2. 修正対象だけでなく、依存するクラス・設定・テストを読み込む
3. 依存関係を芋づる式に追跡する
4. アーキテクチャを内部モデル化する
5. 多段階の変更計画を立てる
6. 整合性を保ったコードを生成する

**だから能力が高い**。プロジェクト全体を理解した上での編集ができる。エンジニアが熱狂的に支持したのは当然だった。

しかし、ここに罠がある。「深く読む」とは、**コンテキストウィンドウへの大量のトークン投入**を意味する。AIにとって読むことは課金そのものだ。

1回の修正タスクで Claude Code が読み込むファイルは20〜50、ときに100を超える。エンジニア1人が1日に何度もこれを実行すれば、月額数千ドルは容易に達する。

ここで決定的な事実が見える:

> **Claude Codeのトークン消費は、コードベースの構造とファイル形式に決定的に依存する。**
> **抽象化が深く、ファイルが分散し、データ形式が重いほど、消費は爆発する。**

Microsoftが扱うのは、世界でも有数の規模のコードベースだ。Windows、Office、Teams、Outlook、Surface — 数十年積層したシステム群。言語構成も多岐にわたる。Windowsのカーネルやファイル形式の中核は**C++**、業務ロジックやAzure統合は**C#/.NET**、Teamsのフロントは**TypeScript/Electron**。

**C++のコードベースはC#/.NETよりさらに重い**。ヘッダー(.h)と実装(.cpp)の分離、#includeの連鎖、テンプレートメタプログラミング、マクロ — すべてがAIの読み込み負荷を増やす。特にUI層までC++で書かれている領域(MFC、ATL、Win32、COM)では、C#/WPFなら1〜2ファイルで済む変更が10〜20ファイルにまたがる。

つまりMicrosoftが直面しているのは、**C++(UI層含む)、C#/.NET、OOXMLドキュメントを扱う製品群が何十年も積層した、AI時代に構造的に最悪の組み合わせ**である。Microsoft自身の歴史的な技術選択の総体が、AI時代に重くのしかかっている。

ここから本稿の中心議論に入る。多くの企業が日常的に使う Office + C#/.NET という組み合わせは、AI時代に構造的にトークン消費が爆発する技術選択である。Microsoftの事例は、その構造的問題が極端に現れた一例にすぎない。

## 3. C#/.NETがAIに強いる構造的負荷

### 3.1 ファイル分散と抽象化の連鎖

C#/.NETの典型的な業務システムは、Clean Architecture、DDD、SOLID原則という「人間にとっての正しい設計」に従って構築されている。だがこれらは、**AIにとっては最悪の設計**である。

1機能を実装するのに以下のファイルが分散する:

- エンティティ、DTO、ViewModel
- リポジトリのインターフェースと実装
- サービスのインターフェースと実装
- APIコントローラ、バリデーション
- AutoMapper設定、DI登録
- Entity Framework設定、appsettings.json

合計10〜15ファイル。それぞれ数百行。

AIが1つのクラスを正確に理解するには、インターフェース→具象クラス→さらに依存先と、芋づる式に展開する必要がある。**この抽象化レイヤーの多重化が、構造的なトークン浪費を生む**。

### 3.2 バリアント型の不在 — C#最大の構造的負債

C#エコシステムがAIに強いる最大の負債は、**バリアント型(判別共用体)のネイティブサポートが長年存在しないこと**である。

業務データには本質的に「複数のパターンのいずれか」という性質がある。受注は通常パターンか特注か返品か。支払いは現金かカードか振込か。F#、Rust、TypeScript、Swiftなら、1つの型定義で簡潔に表現できる。

だがC#では、抽象基底クラスと複数の派生クラス、それらを処理する巨大なswitch文を構築せざるを得ない:

```csharp
public abstract class OrderLine { ... }
public class StandardOrder : OrderLine { ... }
public class CustomOrder : OrderLine { ... }
public class SampleShipment : OrderLine { ... }
public class ReturnOrder : OrderLine { ... }
// 派生クラスごとに別ファイル
```

AIがこの構造を修正するには:
- 基底クラスと全派生クラス(7〜10ファイル)
- switch文を含む業務ロジック(数十ファイル)
- Entity Frameworkの継承戦略マッピング(TPH/TPT/TPC)
- JSONシリアライザのカスタムコンバーター
- ユニットテスト

**1タスクで30〜50ファイル**。バリアント型がある言語なら1ファイルで済む。

Microsoftもこの問題の深刻さを認識している。2026年後半リリース予定の.NET 11(C# 15)プレビューで、長年要望されていた`union`キーワードの導入が予告された。だが、これは新規開発にしか恩恵をもたらさない。**世界中の企業が抱える既存のC#コードベースは、肥大化したクラス階層のまま残り続ける**。

### 3.3 設定地獄

C#/.NETエコシステムは、コード動作に影響する設定ファイルが多い。csproj、appsettings.json、Web.config、マイグレーションファイル、DI登録、ロギング・認証・ミドルウェア設定。AIは全体像を把握しないと正確な修正ができない。

これらすべてが、AIのコンテキストを肥大化させる。

## 4. Officeファイル形式の重さ

### 4.1 OOXMLは構造的に重い

`.xlsx`、`.docx`、`.pptx`は、表面上は構造化データに見えるが、実体は**OOXML(Office Open XML)**という巨大なZIP圧縮XMLの集合体である。

たった1セルに「123」と書かれた`.xlsx`の中身は:

- `[Content_Types].xml`
- `xl/workbook.xml`、`xl/worksheets/sheet1.xml`
- `xl/sharedStrings.xml`、`xl/styles.xml`、`xl/theme/theme1.xml`
- `docProps/core.xml`、`docProps/app.xml`
- その他関係性メタデータ

**「123」一つのためにこれだけのXML処理が必要**。AIにとって、データ1つ取り出すために構造解析で膨大なトークンを消費する。

同じ情報を**Polars/pandas のDataFrame + Parquet**で表現すれば、トークン消費は10分の1以下になる:

```python
import polars as pl

df = pl.DataFrame({
    "product_code": ["A-100"],
    "quantity": [10],
    "unit_price": [1500],
    "delivery_date": ["2026-06-01"]
})
df.write_parquet("orders.parquet")
```

**Parquetは型情報をファイル内部に持つ**。AIはParquet仕様を完全に把握しており、スキーマ説明不要で構造が伝わる。読み込みも`pl.read_parquet("orders.parquet")`の1行で済む。

CSVを選ばない理由も明示しておく。**CSVはスキーマを持たない**。値はすべて文字列で、型は読み込み側の推論次第。`"001"`を文字列か整数1か、`"2026-06-01"`を文字列か日付か、判断が揺れる。業務データのように型の正確性が重要な領域では、CSVは弱い選択肢になる。

**Parquet、SQLite、JSON — これら「スキーマを内包する標準フォーマット」**こそ、AI時代に有利な選択肢である。

### 4.2 横断処理での累積コスト

業務文書は単独で存在しない。SharePointやTeamsには数百〜数千のOOXMLファイルが蓄積されている。

「過去3年の見積書から特定顧客の取引履歴を抽出」のようなタスクは、AIに以下を強いる:

- ファイル探索
- 各ファイルのZIP解凍
- 内部XMLの構造解析
- スタイル定義・メタデータの読み飛ばし
- 本文テキスト抽出

**ファイル1個ごとにOOXML処理のオーバーヘッドが積み上がる**。

旧形式のバイナリ(.doc, .xls)はさらに悪い。AIがネイティブに読めず、専用パーサー経由の変換層がさらにトークンを消費する。

## 5. Office + C#/.NETの相乗効果

C#/.NETとOfficeは、実際の業務システムでは結合している。基幹システムには必ずOfficeファイルの入出力処理がある。受注のExcelインポート、月次レポートのPowerPoint生成、契約書のWord出力。

C#のコードには Excel/Word 操作ライブラリ(`Microsoft.Office.Interop`、`EPPlus`、`ClosedXML`、`OpenXML SDK`)が組み込まれている。

AIがこの結合部を修正するには、**3つの層を横断的に読まされる**:

1. **C#/.NETのコード本体** — エンティティ、サービス、リポジトリ、コントローラ、抽象化レイヤー
2. **OOXMLのデータ構造** — Excelテンプレート、Word/PowerPoint雛形、セル位置、書式
3. **両者をつなぐマッピング層** — セル→C#オブジェクト変換、テンプレート処理

それぞれが独立にトークンを爆発させる要因を持つ。組み合わさることで、**消費は線形的な加算ではなく、指数関数的な相乗効果**となる。

同じ業務処理を Python + 標準フォーマットで実装した場合:

```python
import polars as pl
import sqlite3

# Parquetから読み込み(OOXMLオーバーヘッドなし、型情報込み)
df = pl.read_parquet('orders.parquet')

# 処理(クラス階層なし、dataframeで完結)
returns = df.filter(pl.col('qty') < 0)
customs = df.filter(pl.col('spec_book').is_not_null())

# SQLiteに保存(マッピング層なし)
conn = sqlite3.connect('data.db')
df.write_database('orders', conn, if_table_exists='replace')
```

AIが読むのはこのファイル1つ。スキーマも自明。**Office + C#/.NETで30〜50ファイル必要な処理が、1ファイルで完結する**。

## 6. なぜこれが「構造的問題」なのか

要因を整理する:

| 要因 | AIへの影響 |
|---|---|
| C#/.NETのファイル分散設計 | 1機能で10〜15ファイル読み込み |
| 抽象化レイヤーの多重化 | 依存先を芋づる式に展開 |
| バリアント型の不在 | クラス階層で表現せざるを得ない |
| クラス階層とswitch文の連鎖 | 1タスクで30〜50ファイル |
| 関連設定ファイル | csproj、appsettings、Startup等 |
| OOXMLの構造 | 1セル「123」のために大量のXML処理 |
| 横断処理 | OOXMLオーバーヘッドが線形累積 |
| 旧形式バイナリ | 専用パーサー経由の変換層が必要 |
| Office + C#/.NETの結合 | 3層を横断する読み込み |

**これらは一つのシステム内で同時に発生する**。だから「構造的問題」なのだ。

重要なのは、**これらが「設計の不備」によって生じているわけではない**こと。Clean Architecture、SOLID、DDDはすべて「正しい設計手法」として推奨されてきた。OOXMLも業界標準フォーマットだ。**「正しく設計された」Office + C#/.NETシステムこそが、AI時代に最も効率が悪くなる**。

## 7. 緩和策の限界 — プロンプトキャッシング

AIプロバイダー各社は、トークン浪費問題への緩和策として**プロンプトキャッシング**を導入している。システムプロンプトや変化しない背景情報のKVテンソルを保存し、後続リクエストで再計算をスキップする技術である。

Anthropicは100万トークンあたりのキャッシュ読み取りコストを通常の$3.00から$0.30へ90%削減。あるセキュリティ監査プロジェクトではキャッシュヒット率を7%から84%へ向上させ、全体コストを70%削減した事例もある。

しかし、これは万能ではない。**キャッシュが機能するには、静的なコンテンツをプロンプトの先頭に、動的なコンテンツを末尾に配置する厳密な順序が必要**である。後続リクエストは、キャッシュされたプレフィックスと完全一致した場合にのみヒットする。

C#/.NETのような高度に分散されたシステムでは、タスクごとに読み込むファイルの組み合わせが動的に変わる。**プレフィックスの一貫性を保つことが極めて困難**で、キャッシュミスが頻発する。

つまり、キャッシュ技術の恩恵を受けるためにも、**アーキテクチャ自体が「静的でまとまった構造」を持っている必要がある**。Office + C#/.NETはこの条件を満たさない。

GitHubもこの状況に対応し、2026年6月1日からCopilotの料金体系を「GitHub AI Credits」という従量課金モデルへ移行する。**非効率なコードベースをAIに読ませるコストは、プラットフォーマーからユーザー企業に直接転嫁される**。

## 8. パターン認識から構造推論へ — 認知の転換

エージェント型AIの普及は、エンジニアの働き方を根本から変えつつある。

### Codexが残した教訓

これまでのプログラミングは、本質的に**パターン認識**だった。C#のパターン、WordPressのパターン、Reactのパターン — 既存の「正しい書き方」を覚えて適用する技能だった。ベテランエンジニアとは、多くのパターンを知っている人のことだった。

AIはこのパターン認識を完璧に代行する。特に**OpenAIのCodex**はこの領域で強力で、「リポジトリパターンで書いて」と頼めば定型コードが瞬時に出てくる。**パターンエンジニアの仕事は、Codexに代替される**。

だがCodexを中心に使うと、**最後のバグ取りが極めて難しくなる**。

理由は明確だ。**バグ取りはパターン認識ではなく構造思考の仕事**である。「期待した動作」と「実際の動作」のギャップを構造的に把握し、データの流れを追跡し、なぜそうなるはずだったのかを推論する。これはCodexが得意とする領域ではない。

さらに深刻な問題がある。**Codex(あるいはパターン適用型の人間)は、不整合をテストしにくい場所に押し込む**。

メカニズムはこうだ。Codexは「動くコード」を生成することを目標とする。テストが書かれている箇所では正直に書く。だが矛盾や曖昧な仕様に直面したとき、不整合をどこかに押し込まざるを得ない。そのとき選ばれるのは、**抵抗の少ない場所** — テストの目が届きにくい場所である:

- エラーハンドリング、非同期処理のタイミング
- 状態遷移の中間状態、外部システム境界
- エッジケース(null、空文字、絵文字、タイムゾーン)
- 権限チェックの抜け道、ログ周り、設定値の境界

**バグはここに集まる**。完成時点では誰も気づかない。本番の特殊な条件でしか発症せず、原因追跡が困難になる。

**これはAI特有の問題ではない**。経験の浅い人間も同じ行動を取る。**「品質」の本質は、不整合をテストしにくい場所に隠さない構造を作れるかどうかにある**。

### Claudeの強みは構造思考

Claudeが強みを発揮するのは、まさに「最後の10%」 — 仕様の曖昧さを指摘し、複数ファイルを横断して因果を追い、「このパターン選択自体が間違っているのでは」と問い直す領域である。**バグ取りはAI時代における人間 + Claudeの主戦場**である。

AI時代の役割分担は、こうなる:

- **パターン適用** → Codexに任せる
- **構造を考える** → 人間 + Claudeが担う

時間配分も逆転する。従来の「コード書く80%、バグ取り20%」が、AI時代は「コード書く20%、構造を考える・バグを取る80%」になる。**最初から構造を考えて作らないと、後で取り返せない**。

### ビルダーへの転換は容易ではない

ここに、ソフトウェアエンジニアがビルダーへ転換する難しさがある。

長年パターン認識を磨いてきた人ほど、**パターンの外に出る訓練を受けていない**。「なぜリポジトリパターンを使うのか」「クラスを使わない選択肢はないか」と疑う筋力が萎縮している。

「正しい設計」「綺麗なコード」「ベストプラクティス」 — これらの言葉が、AI時代には**逆効果**になる場面が増える。Clean Architecture、SOLID、DDDを内面化したベテランほど、Pythonのdictと関数だけで完結する素朴なコードを「設計が雑だ」と感じてしまう。**その直感自体が、AI時代には足かせになる**。

興味深い逆説として、**プログラミング未経験者のほうがビルダーに近い場合がある**。既存パターンに縛られていない、「正しい設計」を知らない、現実の困りごとから出発する、AIに任せることに抵抗がない — これらの「素人性」がAI時代には武器になる。

これは「ソフトウェア工学」から「リベラルアーツ(構造を読む力)」へのシフトである。技術学習以上の**認知的変換**を要する。年単位の時間がかかるかもしれない。だが、この転換こそがAI時代の最重要課題である。

## 9. AI時代はPythonに収斂していく — 経済的必然として

ここまで「Markdown + Python + 標準フォーマット」を推奨してきたが、これは思想ではなく**経済的必然**である。Pythonは他言語に対して圧倒的な経済的優位を持つ。理由は5つある。

**1. AIの学習データに最も多く含まれている**
- AIモデル(Claude、GPT、Codex)の訓練データでPythonが圧倒的多数
- AIのPythonコード生成精度が他言語より高い
- 同じタスクでも、Pythonなら少ないやり取りで完成する → トークン消費が少ない

**2. ボイラープレートが少ない**
- 同じロジックを書くのに必要な文字数が少ない
- 型宣言、波括弧、セミコロン、修飾子が不要
- AIが読み込むコンテキスト量が構造的に少ない

**3. dictと標準フォーマットの相性**
- PythonのdictとJSONが透過的
- pandas/polarsのDataFrameとParquet/SQLiteが標準
- データ表現に余計な抽象層が要らない

**4. 1ファイル完結ができる**
- クラス階層なしで関数だけで完結できる
- ヘッダーと実装の分離がない
- import 1行で標準ライブラリが使える

**5. AI時代に既に膨大なエコシステム**
- 機械学習、データ処理、Web、自動化、すべてPython
- AI開発の標準言語(PyTorch、Hugging Face、LangChain、Agent SDK)
- AIが「Pythonでの実装方法」を最も多く知っている

これら5つはすべて「AIのトークン消費を構造的に減らす」方向に働く。同じタスクをPythonで実装するのと、C#/.NETやJava、重いフレームワークを伴うJavaScript/TypeScriptで実装するのとでは、**月額のトークン課金額が一桁違ってくる**。

企業にとって、これは経営判断である。C#/.NETやJavaで基幹システムを抱える企業は、AI保守コストがPythonベースのシステムの数倍から十数倍に膨らむ。**Pythonに移行できない、あるいはシンプルな構成に降りられない組織は、AI時代に経済的に取り残される**。Microsoftの事例は、この経済構造の最も極端な現れである。

## 10. 個人ビルダーは恐れる必要がない

ここまで「嵐」の話を続けてきたが、最後に強調しておきたい。

**個人ビルダーや小規模開発者は、このトークン課金の嵐を恐れる必要がない。**

筆者自身、AIseedプラットフォーム、伝統野菜辞書、自然農データ分析、サーバー管理、執筆 — これらをすべて**Claude MAX $200/月のプラン**で動かしている。十分余裕がある。

ただし、ここで興味深い体験がある。$200プランを使っているのは**主にFlutter開発のため**である。AIseedのフロントエンドはFlutter(Dart)で書いており、Dartは型システムを持ち、Widgetツリーは深い構造を持つ。1画面の修正でも複数Widgetファイルにまたがる。Riverpod や Provider を避けて自己完結型Widgetで書いて負荷を最小化しているが、それでもPythonと比べれば重い。

**一方、Python側の開発(バックエンド、データ処理、Agent SDK、スクリプト類)では、Claude Codeのトークン消費が明らかに軽い**。読み込まれるコンテキスト量が違うのが体感でわかる。Python + 標準フォーマットで書かれたコードは、AIが1〜2ファイル読めば全体を理解できる構造になっている。

つまり、**$100プランで十分な領域(Python + 標準フォーマット)**と、**$200プランが必要な領域(Flutterのような型システムを持つフロントエンド)**がある。両方を使う筆者でも$200で足りる。**Pythonだけで開発する人なら$100で十分**だろう。

大企業が月額数千ドルを払って苦しむとき、個人ビルダーは月額$100〜$200で同じClaude Codeを快適に使い続けられる。**この非対称性こそが、AI時代における個人と小規模の構造的優位**である。

恐れる必要はない。地味な技術選択を続ければいい。**Markdown + Python + 標準フォーマット** — これがAI時代の個人ビルダーの最強の組み合わせである。

## 結論 — 技術選択は経営財務的決断である

AI時代の到来は、技術選択の意味を根本から変えた。

Microsoftが自社基幹部門でClaude Codeを停止し、Uberが予算を早期枯渇させた事象は、3つの冷徹な事実を証明している。

**1. アーキテクチャの不適合は財務的危機に直結する**
Office + C#/.NETという「正しい設計」とされてきた抽象度の高い分散型アーキテクチャは、エージェント型AIのトークン消費メカニズムと決定的に相性が悪い。

**2. コストはモデルの進化では解決しない**
AIが高機能になり、自律的な推論能力を高めるほど、読み込むコンテキスト量は増える。プロンプトキャッシングなどの緩和策は存在するが、分散アーキテクチャの下では効果が限定的である。

**3. 従量課金リスクは企業に転嫁される**
GitHub AI Creditsへの移行に見られるように、非効率なコードベースをAIに読ませるコストは、プラットフォーマーからユーザー企業のバランスシートへ直接転嫁される。

選択肢は既に提示されている。派手な新技術を追うのではなく、**Markdown、Python、スキーマを内包する標準フォーマット(Parquet、SQLite、JSON)の直叩き**という、古くからある標準への**回帰**である。

このシンプルでフラットなアーキテクチャこそが、AIエージェントのコンテキスト消費を最小化し、ユニットエコノミクスを成立させる唯一の合理的な戦略である。

AI時代における「品質」とは、複雑なデザインパターンを駆使することではなく、**AIが読みやすく、キャッシングが機能しやすく、不整合をテストしにくい場所に隠さない、シンプルで透過的な構造**を設計できるかどうかにかかっている。

技術の選択は、もはやエンジニアリングの好みの問題ではない。企業がAI時代を生き残るための最も重要な経営判断である。
