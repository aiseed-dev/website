# 図を描く ── Mermaidで構造を残す

図を描く道具を、Mermaid に変える。

それだけで、図は AI が読み書きできる「構造」になる。PowerPoint や draw.io で描いた図は、ほとんどの場合、画像という墓場に閉じ込められている。テキストで書き直せば、図も同僚になる。

## PowerPoint の図は、コピーした瞬間に死ぬ

ある人が PowerPoint で美しい組織図を描く。それを Word の報告書に貼り付ける。Word ファイルが他の人に渡る。その人が「このボックスの位置を少し変えたい」と思う。

そこで何が起きるか。**ボックスは動かせない**。Word に貼り付けられた瞬間に、図は画像になっている。元の PowerPoint ファイルがどこにあるかは、誰にも分からない。「変更したい」人は、画像の上から新しい図を描き直すしかない。

これは PowerPoint の欠陥ではない。PowerPoint は人間がスライドを作るための道具だ。図はスライドの上で完結することが前提になっている。再利用は二の次の設計である。

しかし、業務で必要な図の本質はスライドではない。本質は構造だ。「A は B に依存する」「処理は X → Y → Z の順で進む」── 図を成り立たせているのはこの関係である。見栄えは表示のための装飾にすぎない。

## Mermaid はテキストで図を書く

Mermaid はテキストだ。記号で図を表現する。

```
graph TD
  A[ユーザー] --> B[Web サーバー]
  B --> C[(データベース)]
  B --> D[キャッシュ]
  C -.->|遅延| B
```

これだけで、4 つのノードと矢印の関係が表現できる。「ユーザーが Web サーバーを叩き、サーバーがデータベースとキャッシュにアクセスする」── ファイルを開いた瞬間に分かる。

Mermaid は「フローチャート」「シーケンス図」「クラス図」「ER 図」「ガントチャート」「状態遷移図」など、業務で必要なほとんどの図を書ける。GitHub・Notion・VS Code・各種 Markdown プレビュー、あらゆる場所がネイティブで Mermaid をレンダリングする。

## バージョン管理できる

PowerPoint ファイルを Git に入れても、差分は分からない。「12 月のスライドと 1 月のスライド、どこが変わった?」── 分からない。バイナリだから、Git には「全部変わった」としか見えない。

Mermaid はテキストだ。Git の差分が普通に出る。「この図の矢印が一本増えた」「ノード名が変わった」── すべて見える。レビューできる。承認できる。元に戻せる。

> 図をテキストで書くだけで、図がコードと同じ品質管理の対象になる。

これは大きい。組織のドキュメントに「図はあるが正確かどうか分からない」「最後に更新したのは誰でいつか分からない」という問題が消える。

## AI が読める

Mermaid のソースは Claude が直接読める。

「この図に新しいノードを追加して」「矢印の方向を反転させて」「シーケンス図に変換して」── 全部、Claude に頼める。出てきた Mermaid をそのまま貼り直すだけで、図は更新される。

PowerPoint の図ではこれができない。図を画像で渡せば、Claude は「絵」として見るが、構造は推測でしかない。Mermaid なら推測の余地はない。

## 何を Mermaid で書くか

業務で出てくるほとんどの図だ。

- システムの構成図
- 処理のフローチャート
- 組織図
- 顧客と注文の ER 図
- プロジェクトのガントチャート
- 状態遷移(アカウント作成 → 認証待ち → 有効化、など)
- API のシーケンス図

「PowerPoint で描いて画像にして報告書に貼る」習慣のある図は、ほぼ全部 Mermaid に置き換えられる。

複雑な絵 ── 写真や手描きのスケッチ、芸術的なダイアグラム ── は別の道具で扱う。しかし「構造を伝える」目的の図は、Mermaid の領域だ。

## 描く必要すらない

ここで一段階先に進む。

Mermaid の記号を覚える必要はない。Claude に「このシステムの構成図を Mermaid で書いて」と頼めば、勝手に書いてくれる。出てきたものを貼って、レンダリング結果を見て、修正したいところを「この矢印を太くして」「キャッシュをデータベースの上に持ってきて」と指示すれば、Mermaid を書き直してくれる。

つまり、**自分で記号を覚える必要すらない**。図にしたい構造が頭の中にあれば、Claude が記号を当てて整形する。

学ぶべきは Mermaid を**読める**ようになることだけ。読めれば修正できる。これも数時間で身につく。

## 10年後も読める

20年前の PowerPoint ファイルを今の PowerPoint で開くと、フォントが置換され、図形の位置がずれる。.ppt 形式は廃止に向かっている。

Mermaid は、ただのテキストファイルだ。Mermaid のレンダラがバージョンアップしても、テキストの記法はほぼ変わらない。10年後も20年後も、テキストエディタがあれば図の元データは読める。

> 図も、構造で持て。

PowerPoint の図は表示の瞬間を飾る。Mermaid の図は時間を超える。

## まとめ

道具を変えれば、図の扱い方が変わる。

PowerPoint から Mermaid へ。一歩で、図が「画像という墓場」から「処理できる構造」に変わる。AI が同僚になる。バージョン管理できる。10年後も読める。
