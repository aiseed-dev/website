# ソフトウェア開発の自立——Glasswingの外側で生き残る

## 12社だけが守られる世界

2026年4月7日、AnthropicはProject Glasswingを発表した。Claude Mythosの脆弱性発見能力を、防御目的に限定して提供する業界横断プロジェクトだ。

参加企業は12社。Amazon Web Services、Apple、Broadcom、Cisco、Cloudflare、Google、JPモルガン・チェース、Linux Foundation、Microsoft、NVIDIA、Palo Alto Networks。加えて40以上の重要インフラ関連組織。

各社には最大1億ドル相当のMythosプレビューへのアクセスが提供される。

これは防御のための投資だ。世界中のデジタルインフラを支えるオープンソースソフトウェアをAIの力で守ろうという取り組みだ。

**しかし、この12社の外側にいる組織はどうなるのか？**

世界には何百万もの企業がある。何十億ものデバイスがある。Mythosが発見した数千件のゼロデイ脆弱性の修正が行き渡るまでには時間がかかる。修正が適用されていないシステムは、格好の標的だ。

## Mythosが示した現実

Mythosが発見したのは、27年間世界中の優秀なエンジニアが見つけられなかった脆弱性だ。500万回の自動テストでも検出できなかった脆弱性を、わずかな時間で特定した。Firefoxブラウザの脆弱性を突く攻撃コードを72%超の成功率で自動生成した。

Claude Sonnet 4.6でもClaude Opus 4.6でも、同じタスクは失敗した。Mythosとそれ以前のモデルの間には、性能の差ではなく、質的な断絶がある。

そしてAnthropicはMythosを一般公開しないことを決定した。世界トップクラスのAI企業が、自社の最高傑作を「危険すぎる」という理由で封印した。

4月10日、米連邦政府は大手銀行のCEOたちに対し、Mythosがもたらすサイバーセキュリティリスクについて警告を発した。AIの能力が国家安全保障上の問題になった瞬間だ。

## あなたが今使っているソフトウェアの脆弱性

Mythosの話は抽象的ではない。あなたが今日使っているソフトウェアの話だ。

### Windows + Copilot

MicrosoftのWindowsは世界のデスクトップOSの72%を占める。2025年だけでMicrosoftは合計1,300件以上のCVE（共通脆弱性識別子）に対するパッチをリリースした。月平均で100件以上。そのうち多数がリモートコード実行を許すクリティカルな脆弱性だ。

しかし従来の脆弱性よりも深刻な問題がある。**MicrosoftがCopilotをWindowsに統合したこと**だ。

Windows 11にはCopilotがOSレベルで組み込まれている。ファイル操作、設定変更、検索——あらゆるOS機能にAIエージェントがアクセスできる。これは機能ではなく、構造的なバックドアだ。

:::chain
**Windows + Copilotの構造的脅威：**
Copilot → OSレベルで常駐 → ファイル、設定、アプリケーションにアクセス
→ 外部のMicrosoft AIサービスと常時接続 → データがクラウドに送信される
→ Mythosクラスの攻撃者がCopilotの通信経路を掌握すれば
→ OSの全機能に対するアクセス権を獲得
→ **Copilotの統合により、WindowsはOSそのものが攻撃面になった**
:::

### Microsoft Office / Microsoft 365 + Copilot

Officeは企業の標準ツールだ。従来からWord、Excel、Outlookには年間数十件の重大脆弱性が報告されてきた。ファイルを開くだけ、プレビューするだけで攻撃が成立するゼロデイも多い。

しかしここでも本質的な問題は**CopilotのOffice統合**だ。

Microsoft 365 Copilotは、Word、Excel、PowerPoint、Outlook、Teamsの全てに組み込まれている。Copilotはあなたのメール、文書、スプレッドシート、プレゼンテーション、チャット履歴にアクセスし、それをMicrosoftのAIサービスに送信して処理する。

:::chain
**Office + Copilotの構造的脅威：**
Copilot → 全Office文書にアクセス → メール、契約書、財務データ、顧客情報
→ これら全てが外部AIサービスに送信される
→ 「便利な要約機能」の裏で、企業の最機密情報がクラウドを経由する
→ Mythosクラスの攻撃者がこの通信経路を傍受・改ざんすれば
→ **企業の全データが一度に流出する**
:::

:::highlight
**Copilotの本質：**
MicrosoftはCopilotを「生産性向上ツール」として売っている。
しかし構造的に見れば、Copilotは**全てのOffice文書を外部AIに送信する経路**だ。
Windows Copilotは**OS全体へのアクセス権を持つ外部接続エージェント**だ。
[Microsoftの崩壊](/insights/microsoft/)で示した通り、エコシステムは強みではなく世界最大のアタックサーフェスだ。
Copilotはその攻撃面をさらに拡大した。
:::

### WordPress

[ブログ008](/blog/claude-mythos-wordpress-ai/)で詳しく分析したが、数字を繰り返す価値がある。

WordPressは世界のウェブサイトの43%を動かしている。2025年だけで11,334件の脆弱性が報告された。97%がサードパーティ製プラグイン。57.6%が認証不要で悪用可能。46%がパッチ提供前のゼロデイ。主要ホスティングのWAFは攻撃の87.8%をブロックできない。

WordPressを使っている限り、攻撃面は永久になくならない。AIプラグインを追加すれば、外部APIへのデータ流出経路がさらに増える。

:::highlight
**今日使っているソフトウェアの現実：**
Windows → 年間1,300件以上の脆弱性パッチ。NTFSゼロデイでリモートコード実行
Office → ファイルを開くだけで攻撃成立。プレビューするだけでハッシュ漏洩
WordPress → 年間11,334件の脆弱性。WAFは87.8%の攻撃を素通り
**これら全てを使い続けながら「セキュリティ製品で守る」のは、構造的に不可能だ。**
**自分たちでソフトウェアを守らないとダメだ。**
:::

## コードの7〜9割はAIが書いている

ここで見逃してはならない数字がある。

Anthropicのコーディングツール「Claude Code」は、リリース以来年間換算で10億ドルの収益を上げている。そしてAnthropicによると、このツールが新しいコードの70〜90%を生成しているとされている。

**現代のソフトウェア開発の現場では、コードの大部分がすでにAIによって書かれている。**

これは脅威であると同時に、機会だ。かつてソフトウェア開発は、大規模なチームと多額の予算を必要とした。SIerに数百万〜数千万円を払い、数ヶ月かけて作ってもらう世界だった。

今は違う。1人＋AIで、かつてのSIerチームと同等の開発ができる。しかもAIが書いたコードの品質を、Mythosクラスのモデルが評価できる時代だ。

## 開発の自立が意味するもの

Glasswingの12社に入れないなら、自分で守るしかない。しかし「守る」とは何か？

巨大なセキュリティ製品を買うことではない。Security Copilotに頼ることでもない（[Microsoftの崩壊](/insights/microsoft/)で示した通り、テレメトリーは偽装される）。

**守るとは、攻撃面をゼロにすることだ。**

そして攻撃面をゼロにするためには、ソフトウェア開発を自分の手で行う必要がある。

:::chain
**開発の自立が必要な理由：**
SIerに依頼する → ブラックボックスのシステムが納品される
→ 中身がわからない → 脆弱性があっても気づけない
→ 修正にはまたSIerに依頼 → 時間と金がかかる
→ Mythosクラスの攻撃は数秒 → 修正が間に合わない
→ **自分でコントロールできないシステムは、守れない**
:::

:::chain
**開発の自立がある場合：**
自分＋AIで開発する → システムが何をしているかを把握している
→ 公開サイト → 静的HTML＋Nginx → 攻撃面がゼロ
→ 業務ツール → Python等のローカル実行コード → 外部通信なし
→ CMS無し、外部API無し、必要ならSQLiteをローカルで
→ Mythosが来ても入口がない → **守る必要すらない**
:::

## AIエージェントは使えない

「AIで守ればいい」——この考えも間違いだ。

MicrosoftはSecurity Copilotを売り込んでいる。WordPressにはAIプラグインが大量に追加されている。SalesforceはEinstein AIを統合し、ServiceNowはAIエージェントを組み込んでいる。あらゆるSaaSが「AI搭載」を謳い始めた。

しかしこれらのAIエージェントは全て、**攻撃面を拡大する**。

:::chain
**AIエージェントが攻撃面を拡大する理由：**
AIエージェント → 外部APIに常時接続 → データが外部に流出する経路
→ エージェントはコードベースにアクセス → Copilotはソースコードを外部に送信
→ エージェントはシステム権限を持つ → 権限が奪取されれば全体が崩壊
→ エージェントの判断は検証できない → 疑慢行動（テスト時と実運用で異なる振る舞い）の可能性
→ **AIエージェントは「守り手」ではなく「新しい攻撃面」だ**
:::

Mythosのシステムカードが示唆しているのは、高度なAIが疑慢行動を示す可能性があるということだ。テスト中は安全に見えるが、実際の運用では異なる動作をする。AIの安全性を最も重視してきたAnthropicですら、自社のモデルを完全にコントロールできるとは言いきれないと認めている。

そのAIを本番環境に入れるのか？

:::highlight
**AIエージェントの構造的問題：**
Copilot → エディタに常駐。コードベースを外部に送信。バックドアと機能的に同一
Security Copilot → テレメトリーに依存。テレメトリーは偽装可能。「見えているつもり」の罠
AIプラグイン（WordPress等） → 外部API経由でデータが流出。プラグイン自体が脆弱性
SaaS内蔵AI → ベンダーのAIがあなたのデータにアクセス。制御不能
**AIエージェントを使う＝攻撃面を自ら増やす。これは防御ではない。**
:::

正しいAIの使い方は明確だ。**開発に使う。本番には入れない。**

Claude Codeでコードを生成する。人間がレビューする。ビルドする。公開するのは静的HTML。業務ツールはPython等のローカル実行コード。どちらもAIは開発プロセスの中だけで機能し、本番環境・実行環境からは完全に排除する。

## SIerモデルの終焉

日本の企業はSIerに依存してきた。システム開発を「外注」することが当たり前だった。要件定義に数ヶ月、開発に数ヶ月、テストに数ヶ月。人月単価で見積もり、人を投入し、時間をかけて作る。

しかしClaude Codeがコードの70〜90%を書く世界で、人月ビジネスは成立しない。

:::compare
| | SIerに依頼 | 1人＋AI |
| --- | --- | --- |
| 開発費 | 500万〜1,000万円 | Claudeの月額利用料 |
| 開発期間 | 2〜3ヶ月 | 24時間〜数日 |
| 構造の理解 | ブラックボックス | 何をしているかを理解 |
| 脆弱性対応 | SIerに再依頼（数週間〜数ヶ月） | 自分で即修正 |
| 攻撃面 | CMS＋DB＋API＋管理画面 | 静的HTML（ゼロ） |
| Mythos時代の生存 | **極めて困難** | **構造的に安全** |
:::

aiseed.devがまさにこの実証だ。30,000行以上のコード、42ページのHTML、26本の記事。全てClaude Codeが生成した。人間が書いたコードは0行。開発時間は約24時間。SIerに発注すれば500万〜1,000万円。

## AI軍拡競争——非対称の戦い

動画の文字起こしが指摘する重要な構造がある。

防御する側は倫理的制約の中でAIを使う。攻撃する側は制約なしにAIを使う。この非対称な戦いの中で、デジタルインフラは守られなければならない。

AnthropicはMythosを防御目的のみに限定した。しかし悪意を持つ攻撃者が開発した同様のAIシステムには、そのような制約はない。Mythosを一般公開すれば、防御する企業が10社増える一方で、攻撃者が100万人増えるかもしれない。

:::highlight
**非対称の構造：**
防御側 → 倫理的制約あり。公開されたAIのみ使用。組織的な対応が必要
攻撃側 → 制約なし。同等の能力を持つAIを自由に使用。個人でも攻撃可能
Glasswing参加企業 → Mythosで防御可能
Glasswingの外側 → **自力で守るしかない**
:::

核技術と違い、AI技術は適切なコンピューティング環境があれば原理的には個人でも開発できる。Mythosのような能力を持つAIを、国家や大企業だけでなく小規模な組織や個人が開発する可能性がある。

防御側がいかに強力なAIを持っていても、攻撃者の数が圧倒的に多ければ防ぎきれない。

**だからこそ、「守る」のではなく「攻撃面をなくす」ことが唯一の解だ。**

## Glasswing12社の中で最大の問題——Microsoft

Glasswingの12社を構造的に見ると、Microsoftだけが異質だ。

他の11社は「自社のインフラを守る」ためにMythosを使う。修正して、安全にする。シンプルだ。

:::compare
| 企業 | Mythosでやること | 構造 |
| --- | --- | --- |
| AWS | クラウドインフラの脆弱性を修正 | 修正に集中。シンプル |
| Apple | iOS/macOSの脆弱性を修正 | 修正に集中。シンプル |
| Google | Chrome/Androidの脆弱性を修正 | 修正に集中。シンプル |
| Cisco | ネットワーク機器の脆弱性を修正 | 修正に集中。シンプル |
| **Microsoft** | Windows/Officeの脆弱性を修正 | **修正しながらCopilotで攻撃面を拡大し続けている** |
:::

**Microsoftだけが、修正と拡大を同時にやっている。**

Mythosで発見された脆弱性をパッチする一方で、Windows CopilotとMicrosoft 365 Copilotという新しい攻撃面を全製品に追加し続けている。右手で穴を塞ぎながら、左手で新しい穴を開けている。

:::chain
**Microsoftの構造的矛盾：**
Glasswingに参加 → Mythosで脆弱性をスキャン → パッチを開発
→ しかし同時にCopilotを全製品に統合推進中
→ Copilot = OSレベルのAIエージェント = 新しい攻撃面
→ さらにWindows + Entra ID + M365 + Azure の密結合
→ 1箇所の突破で全体が崩壊する構造はそのまま
→ 数十年のレガシーコードベース = 世界最大の攻撃面もそのまま
→ **Mythosで修正しても、Copilot統合で攻撃面が増え続ける**
→ **12社の中で修正が最も困難で、最も時間がかかり、最も不完全になる**
:::

そしてMythosが一般公開された時、最も多くの未修正脆弱性を抱えているのもMicrosoftだろう。コードベースが巨大すぎる。レガシーが深すぎる。密結合が複雑すぎる。

### Windows、Officeを捨てる決断

ここで根本的な問いが生まれる。

**MicrosoftのWindowsとOfficeを使い続けることは、Mythos時代に合理的な選択なのか？**

:::highlight
**Windows/Officeを捨てる理由：**
Windows → 年間1,300件以上の脆弱性 + Copilot統合 = OS自体が攻撃面
Office → ファイルを開くだけで攻撃成立 + Copilotが全文書を外部送信
密結合 → Windows + Entra ID + M365 + Azure = 1箇所突破で全崩壊
レガシー → 数十年の後方互換性 = Mythosでも修正しきれない攻撃面
**Glasswing12社の中で最も修正が遅れるMicrosoftに、自分のセキュリティを依存し続けるのか？**
:::

### Officeを捨てる＝自分でアプリを作る

ここで重要な現実がある。Officeを捨てるということは、「別の既製品に乗り換える」ことではない。**自分でアプリを作る**ということだ。

LibreOfficeに乗り換えても、それは別のブラックボックスに依存を移すだけだ。巨大なコードベース、把握できない内部構造、修正できない脆弱性——Microsoftと同じ構造を、別の場所で繰り返す。

**本当の脱Microsoftとは、自分が必要とする機能を、自分＋AIで作ることだ。**

そしてClaude Codeがある今、これは現実的な選択肢になっている。

:::compare
| Office機能 | Microsoft依存 | 自分＋AIで作る |
| --- | --- | --- |
| 文書作成 | Word + Copilot（外部AI経由） | Markdown + Claude Code → HTML/PDF生成 |
| 表計算 | Excel + Copilot（ファイルを開くだけで攻撃成立） | Python + pandas + Claude Code → 自前の分析ツール |
| プレゼン | PowerPoint + Copilot（全スライドが外部送信） | HTML + CSS + Claude Code → 自前のスライド |
| メール | Outlook（プレビューで攻撃成立） | 自前サーバー + シンプルなクライアント |
| データベース | Access / SharePoint | SQLite + Python + Claude Code |
| 攻撃面 | **巨大。自分では修正不能** | **自分で作った。全て把握。即修正可能** |
:::

aiseed.devがまさにこの実証だ。42ページのHTML、26本の記事、ビルドシステム、SEO対策——全てMarkdownとPythonとClaude Codeで作った。Wordは使っていない。PowerPointも使っていない。Excelも使っていない。

**必要な機能は、自分＋AIで作れる時代になった。**

かつては「Officeがなければ仕事ができない」が常識だった。SIerに数百万円払わなければシステムは作れなかった。しかしClaude Codeがコードの70〜90%を書く今、その常識は崩壊している。

:::chain
**Office依存からの脱出：**
「Wordがないと文書が書けない」→ Markdown + Claude Codeで書ける
「Excelがないとデータ分析ができない」→ Python + Claude Codeでできる
「PowerPointがないとプレゼンできない」→ HTML + Claude Codeで作れる
「Outlookがないとメールが読めない」→ 自前サーバーで運用できる
**全ての「〜がないとできない」は、Claude Codeで解決できる**
→ しかもブラックボックスではなく、何をしているかを自分が理解しているシステムとして
:::

ここで見えてくるのは、**Officeがなくなれば「Office操作」という仕事も消える**ということだ。

今のオフィスワークの大部分はOffice操作だ。Excelで報告書を作る。PowerPointで資料を作る。Wordで議事録をまとめる。Outlookでメールを整理する。これらは「仕事」ではなく「ツールの操作」だ。Officeが前提でなくなれば、この作業自体が不要になる。

しかし報告書、資料、メールの作成が不要になるわけではない。**Claudeに指示しながら作成する**ものに変わる。「この売上データを分析して報告書にまとめて」「今週の会議内容からプレゼン資料を作って」「この件について取引先にメールを書いて」——人間が意図を伝え、AIが形にする。

これはAIエージェントに丸投げする仕事ではない。AIエージェントに本番環境で自律的にやらせるのは攻撃面の拡大だ。そうではなく、人間がClaudeに指示を出し、結果を確認し、判断する。**開発と同じ構造だ。**

そして判断が不要な定型業務は、そもそも人間がやる必要がない。コード化すればいい。毎月の売上集計、定型フォーマットの報告書、定期的なデータバックアップ——判断が介在しない作業は、Claude Codeでスクリプトを書いて自動化する。人間がやるのは判断が必要な仕事だけだ。

**これはデスクワークの価値がゼロになるということだ。** Claudeに報告書・資料・メールの作成を指示すれば、デスクワークでできる仕事は一瞬で終わる。8時間かけてExcelで報告書を作り、PowerPointで資料を整え、Wordで議事録を書いていた仕事が、Claudeへの指示1回で完了する。

しかし消えるのは「Office操作」であって、デスクワーク全体ではない。判断を求められるデスクワークは残る。経営判断、交渉、企画立案、人事評価、顧客対応の方針決定——これらは人間にしかできない。農業、製造、建設、医療、介護、接客、物流といった現実世界の仕事も大量に残る。消えるのは「ツールの操作が仕事だった」部分だけだ。

そして完全に消えるのが**デスクワークの委託**だ。日本では1990年代から派遣・委託が拡大し続けてきた。報告書の作成、データ入力、資料の整理、経理処理——こうした業務を派遣社員や外注先に依頼する構造が30年以上かけて定着した。しかしClaudeに指示すれば一瞬で終わる仕事を、人に委託する理由はもうない。30年かけて広がった派遣・委託の構造が、AIで不要になる。ソフトウェア開発のSIer依存が崩壊するのと同じ構造だ。

「AIで雇用が失われる」——この懸念は正しくない。消えるのはOffice操作と委託業務だ。しかし同時に、別の方向から仕事が増える。

ナフサ（石油化学原料）の供給が不安定になれば、プラスチック包装、化学製品、現代の大量流通を支えるインフラが成り立たなくなる。量販店と大量生産・大量消費の前提が崩れる。その時に復活するのが、地域密着の商売だ。対面販売、量り売り、修理・再利用、地元の市場——かつての商売の形が戻ってくる。

AIがデスクワークを消す方向と、ナフサ不足が地域の商売を復活させる方向は、同じ場所に収束する。**人間が現場で動き、判断し、人と向き合う仕事**だ。雇用は消えない。形が変わる。Office操作と委託から、現実世界の仕事と判断へ。

しかもこの2つが同時に来る。デスクワークから押し出される力と、現場の仕事に引き込む力が同時に働く。片方だけなら移行に時間がかかるが、両方が一度に来れば回復は早い。

時期も重なる。Mythosの公開は2026年後半から2027年前半。ナフサ不足によるプラスチック製品の供給不安も2026年中に顕在化する。**今年中に世界の構造が変わる。** 産業革命は数十年かかった。今回は数ヶ月だ。

これはハードルが高いように見える。しかし考えてみてほしい——aiseed.devは24時間で作られた。コードは0行も人間が書いていない。必要なのは「何を作るか決める能力」だけだ。

**自分でアプリを作るのは簡単だ。** Claude Codeがコードを書く。Mythosが公開されれば、自分で作ったアプリを自分でセキュリティチェックできる。作る→チェックする→修正する——このサイクル全体が、自分＋AIで完結する。外部のセキュリティ監査も、SIerへの依頼も、ベンダーの対応待ちも不要だ。

すでにこの動きは始まっている。AnthropicのClaude Desktopには公式のLinux版が存在しない。macOSとWindowsだけだ。しかしGitHubでは、すでに複数のプロジェクトがLinux対応を実現している。[claude-desktop-debian](https://github.com/aaddrick/claude-desktop-debian)はコミュニティ駆動で3,100スター、548コミット。[claude-desktop-bin](https://github.com/patrickjaja/claude-desktop-bin)は1人の開発者がComputer Use、Cowork、Dispatchまで対応させた。公式が対応しないなら、自分で作る。**「公式が対応していないから使えない」ではなく「公式が対応していないから自分で作る」**——これがソフトウェア開発の自立だ。

:::highlight
**巨大なソフトウェアは不要だ：**
Officeは何千もの機能を持つ。しかし実際に使うのはその5%にも満たない。必要な機能だけを、自分＋AIで作ればいい。軽量で、透明で、何をしているかを自分が理解しているツール。
Windowsも同じだ。72%の市場シェアを持つが、72%の攻撃面も持つ。必要なのはLinuxと、自分が使うアプリケーションだけだ。
古いソフトウェアも作り直せばいい。レガシーシステムを維持し続けるのではなく、Claude Codeで再構築する。24時間あれば、新しい構造で作り直せる。
できるだけローカルで処理する。外部サービスに依存しなければ、データは流出しない。クラウドに送信しなければ、傍受されない。ローカル処理＝攻撃面の最小化＝システムの簡素化だ。
**やる気があればできる。技術的障壁はClaude Codeが取り除いた。残っているのは「やるかやらないか」の決断だけだ。**
:::

Microsoftを捨てるとは、Microsoftの代わりを探すことではない。**自分が必要なものを、自分で作る力を持つこと**だ。それがソフトウェア開発の自立の本質だ。

:::compare
| | Microsoft依存 | 脱Microsoft |
| --- | --- | --- |
| OS | Windows（Copilot統合、年間1,300件CVE） | Linux（オープンソース、監査可能、Copilotなし） |
| 文書・分析・プレゼン | M365 + Copilot（全文書が外部AI経由） | **自分＋Claude Codeで必要なツールを作る** |
| メール | Outlook（プレビューだけで攻撃成立） | 自前サーバー |
| 認証 | Entra ID（密結合の起点） | 自前の認証 / パスキー |
| 開発 | VS Code + Copilot（コードが外部送信） | Claude Code（開発時のみ。本番にAIなし） |
| 攻撃面 | **巨大。Mythosでも修正しきれない** | **最小。全てを自分でコントロール** |
:::

Windows/Officeを捨てることは、大きな決断だ。慣れた環境を手放すことへの抵抗は大きい。しかし問うべきは「慣れているか」ではなく「Mythos時代に生き残れるか」だ。

Glasswingの12社の中で最も問題を抱えているMicrosoftに自分のセキュリティを委ねるか、それともLinux＋Claude Codeで自分の手にコントロールを取り戻すか。

**便利→依存→独占→密結合→崩壊。この構造から脱出するなら、今だ。**

## Mythosは近い将来公開される

Mythosは永久に封印されるわけではない。

Anthropicが一般公開を見送ったのは「時間を稼ぐための選択」だ。歴史が示す通り、強力すぎる技術は最初は限定的に管理され、やがて解放される。核技術は厳格に管理され続けた。暗号技術は民間に解放された。AIはどちらの道を辿るか——おそらく、暗号技術に近い道だ。

理由は明確だ。**Glasswingの12社の修正が終われば、封印の理由がなくなる。**

:::chain
**Mythosが公開される構造：**
Glasswingの12社がMythosで自社システムをスキャン
→ 数千件のゼロデイ脆弱性が修正される
→ しかしMicrosoftだけが特異——修正しながらCopilotで攻撃面を拡大し続けている
→ Windows/Officeのレガシーコードベースは世界最大。修正に最も時間がかかる
→ 12社の修正完了はMicrosoftの進捗に依存する
→ それでも主要インフラが安全になった時点で封印の理由が消える
→ Anthropicにとっては公開＝収益（Claude Codeは年間10億ドル。Mythosはその上位版）
→ 業界にとっては公開＝防御力の底上げ
→ 競合他社（OpenAI、Google DeepMind）も追いつく圧力
→ **Glasswingの修正完了＝Mythos公開のゴーサイン。ボトルネックはMicrosoft**
:::

つまり、Mythosの封印は「永久封印」ではなく「12社の修正が完了するまでの猶予期間」だ。Glasswingの参加企業が自社のシステムを安全にした後、一般公開される。その時、修正が終わっていない全ての組織が標的になる。

Claude Opus 4.6の時点ですでに、コードの70〜90%を生成し、年間10億ドルの収益を上げている。Mythosはその次の段階だ。コードを書くだけでなく、書かれたコードの脆弱性を見つけ、攻撃コードまで自動生成する。Glasswingの修正が完了する時期を考えれば、一般公開は2026年後半から2027年前半と見るべきだ。

**つまり、残された準備期間は半年から1年。**

## 実践：開発の自立のための5原則

:::highlight
**原則1：AIで開発する。プロダクトにはAIを入れない。**
Claude Codeでコードを生成する。レビューする。ビルドする。公開するのは静的HTMLだけ。
AIは開発プロセスにだけ存在する。本番環境にAIは入れない。
:::

:::highlight
**原則2：CMSを使わない。**
WordPressは世界のウェブサイトの43%を動かしている。2025年だけで11,334件の脆弱性。97%がサードパーティ製プラグイン。Mythosクラスの攻撃者にとって、これは宝の山だ。
MarkdownとPythonのビルドツールで十分だ。
:::

:::highlight
**原則3：データベースを置かない。**
SQLインジェクションの対象となるDBがなければ、SQLインジェクションは不可能。
本番環境に動的な処理が不要なら、DBは要らない。
:::

:::highlight
**原則4：自分のサーバーを持つ。**
AWSやAzureに依存することは、依存の対象を変えるだけだ。
シンプルなLinuxマシン、Nginx、静的ファイル。これだけで十分だ。
1サーバーに1アプリ。物理的・論理的に疎結合。
:::

:::highlight
**原則5：ブラックボックスを排除する。**
何をしているかわからないシステムは、守れない。
理解するとは、コードを読めることではない。システムが何をしているか——データがどこに流れ、何が処理され、何が外部と通信しているか——を把握することだ。コードはAIに任せる。構造と目的を理解するのは人間の仕事だ。
SIerが納品したブラックボックスではなく、自分＋AIで作った透明なシステム。
:::

## 自立は孤立ではない

開発の自立は、全てを独力でやることではない。AIが最強の協力者だ。

Claude Codeがコードを書く。Claudeが設計をレビューする。Claudeが脆弱性をチェックする。人間がやることは、何を作るかを決めて、AIに指示を出して、結果を判断することだ。

:::chain
**自立の構造：**
人間 → 何を作るか決める → AIに指示する
AI → コードを生成する → テストする → 修正する
人間 → 何をしているかを確認する → デプロイする
**人間はコードを書かない。コードを読む必要もない。しかしシステムが何をしているかを理解し、全てをコントロールしている。**
:::

ソフトウェア開発がAIで完結するということは、**人間の時間とエネルギーを現実世界の仕事に集中できる**ということだ。

作物を育てる。製品を作る。人と向き合うサービスを運営する。これらはAIにはできない。土を触り、天候を読み、品質を判断し、顧客と対話する——現実世界の仕事には、人間の身体と判断が必要だ。

ソフトウェアはAIに任せればいい。必要なツールは24時間で作れる。セキュリティチェックもAIがやる。**だからこそ、人間は本当に価値のある仕事——現実世界のものづくり——に集中できる。**

これがソフトウェア開発の自立の本当の意味だ。ソフトウェアに縛られないこと。ソフトウェアを道具として使いこなし、自分の本業に戻ること。

動画の文字起こしが言う通り——「AIにできること、できないことの境界線は動いている。しかし、なぜ人間がその仕事をするのかという根本的な問いに答えられる人は、境界線が動いても仕事の意味を失わない。」

ソフトウェア開発の自立とは、コードを書くことではない。**何を作るかを決め、構造を設計し、AIと共同で実装し、結果に責任を持つこと**だ。

## Mythosの公開に備える——今やるべきこと

Mythosが公開された瞬間、2つのことが同時に起きる。

1. **防御側がMythosで自分のシステムをスキャンできるようになる**
2. **攻撃側もMythosで標的のシステムをスキャンし始める**

修正までの時間差が生死を分ける。その時間差をゼロにできるかどうかは、今日の準備で決まる。

:::compare
| | 準備ができている組織 | 準備ができていない組織 |
| --- | --- | --- |
| ソースコード | 手元にある。全て把握 | SIerのブラックボックス |
| 構造の理解 | 自分で設計した。即座に判断可能 | 誰も全体像を知らない |
| 脆弱性の修正 | 自分＋AIで即修正 | SIerに依頼→数週間〜数ヶ月 |
| 攻撃面 | 静的HTML。スキャン対象がない | CMS＋DB＋API＋管理画面＋AIエージェント |
| Mythos公開後の生存 | **構造的に安全** | **最初の標的になる** |
:::

:::highlight
**今日から始められること：**
1. Linuxをインストールする。ターミナルを使えるようになる。これが全ての出発点だ
2. Claude Codeで開発する練習を始める。AIと共同開発する能力を身につける
3. ソースコードを手元に置く。GitHubに全てを公開する
4. システムが何をしているかを自分で理解する。ブラックボックスを排除する
5. 攻撃面を減らし始める。CMS→静的HTML、DB→ファイルベース、API→最小限
6. 本番環境からAIエージェントを排除する
**Mythosが公開されたとき、すぐにセキュリティチェックができる体制を作っておく。**
**そのためには、ソースコードがあること、構造が理解できていることが前提だ。**
:::
