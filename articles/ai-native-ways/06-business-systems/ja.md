# 業務システムと付き合う ── 既存資産を活かしつつAIで補う

業務システムを担当するあなたへ。

「Java も C# も捨てて、全部 Python に書き直せ」と言われると、現実的でないと感じる。当然だ。10年・20年動いている基幹システムには、業務知識の集積がある。ドキュメントに書かれていない例外処理がある。書き直すリスクは、現状維持のリスクよりはるかに大きい。

書き直す必要はない。**境界を引き直す**だけでいい。中身は今まで通り Java や C# のまま、新しく付け加える機能は Python と AI で書く。境界をデータ(JSON / CSV)で繋ぐ。

## 全部書き直すは罠だ

「レガシーシステムを Python で書き直す」プロジェクトの大半は、失敗する。

理由は単純だ。古いコードには、ドキュメントには書かれていない業務ルールが埋め込まれている。「7 月の請求は 10 日締めだが、お盆休みを考慮して翌月 5 日まで延長する」── このルールは、誰も覚えていない。コードの中だけにある。

書き直すと、このルールが落ちる。落ちたことに気づくのは数ヶ月後、何かが間違って請求された時だ。

**コードは仕様書の代わりに動いている**。書き直しは、仕様書ごと書き直すことに等しい。それは新しいシステムを一から作るより難しい。

## 境界を引き直す

新しい付き合い方は、境界を引き直すことだ。

既存のシステムは、これまで通り Java や C# で動かす。新しい機能は、既存システムから出てくるデータ(CSV や JSON)を入力にして、Python と Claude で書く。新しい機能の出力も、CSV や JSON で既存システムに渡す。

```
既存基幹システム (Java/C#)
       │
       │  CSV / JSON エクスポート
       ↓
新しい処理層 (Python + Claude)
       │
       │  CSV / JSON インポート
       ↓
既存基幹システム (Java/C#)
```

既存システムの内部には触らない。**境界面のデータだけ AI ネイティブにする**。

これなら、業務知識を失わずに、新機能の開発速度を 10 倍にできる。リスクは小さく、効果は大きい。

## データソースとして扱う

既存システムを「データソース」として扱う。

Java で書かれた基幹システムから、毎日 CSV や JSON を吐き出させる。データベースを直接読んでもいい。XML の API があるなら、それを Python から叩く。

「既存システムを直接呼び出す」のではない。「既存システムが出すデータに対して新しい処理をかける」のだ。**境界がデータなので、AI が触れる**。

## 例: 月次の異常検知

例えば、毎月の売上データから「異常な動き」を検知する仕組みが欲しい。

これまでの方法:
1. 基幹システムを修正する案件として起案する
2. 設計書を書く
3. レビューを受ける
4. Java で実装する
5. テストする
6. 本番リリース申請を出す
7. 数ヶ月後に動く

新しい方法:
1. 基幹システムから売上 CSV を毎日エクスポートする(ジョブを既存に一つ足すだけ)
2. Python スクリプトで異常検知をかける(Claude に書かせる)
3. 結果を CSV か Slack 通知で出す
4. 翌日には動いている

基幹システムは触らない。**境界面に新しい層を一つ足すだけ**。

異常検知の精度を上げたければ、Python スクリプトをチューニングする。やめたければ、ジョブを止めるだけ。基幹システムへの影響はゼロだ。

## 業務知識を Markdown に出す

既存システムには業務知識が埋まっている。それを少しずつ Markdown に出していく。

Java のコメント、SQL のクエリ、運用手順書、過去の障害報告 ── これらを Claude に渡して、Markdown で整理してもらう。「このコードは何をしているか」「どんなルールが埋め込まれているか」を、人間が読める文書にする。

完璧でなくていい。**業務知識が AI が読める形で増えていく**ことが重要だ。次に同じ業務を扱うとき、Claude は過去の Markdown を参照できる。蓄積する。

これは、いつかシステムを置き換えるときの保険にもなる。仕様書がコードと別に存在していれば、置き換えのリスクは下がる。

## SQL は今でも強い

業務システムの中心はデータベースだ。SQL は 50 年前からある言語で、これからも 50 年動く。

Python から既存データベースを SQL で叩く。Claude に「この要件に合う SQL を書いて」と頼めば、複雑なクエリも数秒で出てくる。**SQL を覚える必要は、ほとんど無い**。読めれば十分だ。

これは古い PL/SQL や T-SQL でも同じ。Claude は両方書ける。

## ベンダーロックインから抜ける

Oracle、SAP、Salesforce、Microsoft の業務製品 ── これらは「製品」を売っているのではなく、「製品を使い続けないといけない状況」を売っている。

抜ける道は二つだ。

**一: 一気に乗り換え**

これはほぼ失敗する(前述のとおり)。

**二: 境界を引き直して、徐々に重心を移す**

新機能は外側で書く。データを CSV / JSON で吐かせる。製品が握っているのは「過去のデータと、過去のルール」だけになる。新しい価値は外側で生まれる。

数年経つと、製品の重要度が下がる。契約更新のときに「もう要らないかも」と判断できる材料が揃っている。**ベンダーから自分の側に主導権を取り戻す**。

> 既存資産は壊すな。境界を引き直して、外側で AI と動け。

## まとめ

業務システムは書き直すな。残せ。

新しい仕事は、境界の外側で Python と Claude で書く。データは CSV と JSON で繋ぐ。業務知識は少しずつ Markdown に出していく。

これで、既存資産の業務知識を失わずに、開発速度を 10 倍にできる。リスクが小さく、効果が大きい。
